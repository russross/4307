### Flat Files Approach (CSV/JSON) and Simple Python-Style Processing

*   Explained typical usage workflow:

### Why Flat Files Break Down: Integrity, Complexity, Scale, Concurrency, Durability

*   Highlighted “what goes wrong” with flat files despite their simplicity
*   Data integrity problems:
    *   Inconsistent spelling of artist names across records
    *   Invalid values in typed fields (e.g., a year that is not numeric)
    *   References to artists in the album data that do not exist in the artist data
    *   Renaming problems (if a band name changes, you must update it everywhere manually)
*   Modeling/complexity problems:
    *   Multi-artist albums don’t fit well in a single “artist” field or hierarchical layout
*   Implementation and scale problems:
    *   With millions of artists/albums, files get large and slow to load/rewrite
    *   Sharing the data with other applications leads to duplicated “integrity checks” and “optimization tricks” in multiple codebases
*   Concurrency and corruption risks:
    *   Two programs reading/editing/writing the same file can overwrite each other (“last writer wins”)
*   Durability/availability concerns:
    *   Crashes mid-update can lose or corrupt data
    *   Replication/copying for availability requires manual synchronization across copies
*   Concluded that flat files shift responsibility for all these issues onto the programmer, and systems often “outgrow” the original simple use case

### What a Relational DBMS Provides “Out of the Box”

*   Introduced relational database management systems (RDBMSs) as more than a data model: an engineered system providing standard guarantees
*   Data integrity enforcement:
    *   Field constraints (e.g., year must be in a reasonable range, not in the future)
    *   Referential integrity (album referencing an artist must reference an existing artist)
*   Handling complexity via relational modeling:
    *   Emphasized normalized design as a way to represent complex data cleanly and grow over time
    *   Noted extreme-scale exceptions (Google/Meta), but argued typical apps aren’t at that scale
*   Query language and query execution:
    *   Separating the query from the implementation: you write what you want; the DB figures out efficient execution
    *   Stressed the value of ad hoc querying versus needing to deploy new programs for each question (contrast with key-value stores)
*   Durability and transaction support:
    *   Introduced “decades of engineering” in databases to solve concurrency/durability/optimization problems

### ACID Transactions: Meaning and Intuition

*   Introduced ACID transactions as the “gold standard” for RDBMS durability and correctness under concurrency and crashes
*   Expanded the acronym:
    *   Atomicity
    *   Consistency
    *   Isolation
    *   Durability
*   Atomicity:
    *   A group of changes is “indivisible”: either all happen or none happen
    *   If a crash occurs mid-transaction, the system returns to a state “as though nothing had happened”
*   Consistency:
    *   The DB prevents operations that violate constraints (e.g., type/range checks, referential integrity)
*   Isolation:
    *   Concurrent users/applications shouldn’t create inconsistent results or overwrite each other improperly
    *   Typical app development relies on “open a transaction and let the database handle it”
*   Durability:
    *   Committed data should survive crashes; crashes should not corrupt the database
*   Compared to durability tactics in file-based tools (e.g., editors writing a new file then atomically renaming it over the old one) and noted how tedious it is to reimplement repeatedly

### Separating Modeling/Indexing from Queries; Scaling Advice and Industry Examples

*   Re-emphasized the key idea: separate data modeling and indexing from querying
    *   Store data in a well-modeled format
    *   Ask arbitrary queries; DBMS takes responsibility for efficient execution (with help from indexes)
*   Revisited prior “Twitter/Lady Gaga” style example:
    *   Worst case: celebrity tweets require “fan-out” to millions of timelines; can cause ordering/latency issues
    *   Mentioned strategies like denormalization and optimization for specific access patterns
    *   Argued many systems start with a normal RDBMS (e.g., PostgreSQL) and evolve as needed
*   Quoted engineering advice (attributed to Jeff Dean/Google):
    *   Don’t design for “infinite scalability”
    *   Designing beyond ~100x current dataset is often solving the wrong problems and adding overhead
*   Google example (inverted index / “reverse index”):
    *   Word → list of document IDs; queries intersect sets
    *   Scaling through replication, sharding hot terms, parallel execution
    *   Later design shift: enough machines to keep indexes in memory, changing earlier design assumptions
*   Takeaway: grow based on real data/experience; programmers are poor at predicting future constraints
*   Personal conclusion: relational databases are extremely successful and hard to “fight”

### Relational Model Terminology: Tuples, Attributes, Relations, and `NULL`

*   Noted two vocabularies: formal relational-model terms vs. SQL/DB practitioner terms
*   Defined mappings:
    *   Tuple ≈ row/record
    *   Attribute ≈ column
    *   Relation ≈ table (unordered set of tuples with the same attributes)
*   Atomic/scalar values:
    *   In pure relational model, tuple values are atomic (not lists)
    *   Acknowledged real databases sometimes relax this (e.g., storing serialized lists in a string), which is a form of denormalization / normalization violation
*   `NULL`:
    *   Special value in every domain (unless disallowed) representing “no value”
    *   Not equivalent to `0` or empty string; has special rules in SQL

### Keys: Primary Keys, Natural vs. Surrogate Keys, UUIDs; Hash Analogy

*   Primary key:
    *   Uniquely identifies a single tuple/row in a relation
*   Natural keys:
    *   Keys derived from existing real-world attributes (e.g., band name)
    *   Not always truly unique in practice; discussed name collisions (multiple bands with same name)
    *   Mentioned actors’ unions/guild practices requiring unique credited names (use of middle initials, etc.)
*   Surrogate (synthetic) keys:
    *   Add an integer `id` purely to uniquely identify rows
    *   Common and convenient, sometimes overused (especially due to tooling/ORM assumptions)
    *   Many DBMSs auto-generate integer primary keys on insert
*   Distributed-system issue:
    *   A shared global counter can be a bottleneck at large scale/distribution
*   UUIDs:
    *   Increasingly common alternative; generated from timestamp + randomness, designed for extremely low collision probability without coordination
    *   Discussed “finite but astronomically large” ID space; practically non-problematic if using standards rather than a homemade scheme
*   Analogy to cryptographic hashes:
    *   Mentioned `MD5` (128-bit) and `SHA-256` (256-bit)
    *   Explained “broken” meaning: collisions have been found/constructed; security community migrates to newer algorithms

### Foreign Keys and Relationship Modeling (One-to-Many and Many-to-Many)

*   Foreign key definition:
    *   A set of attributes in one relation referring to the primary key of another relation
*   Artists/albums example:
    *   `albums.artist_id` references `artists.id`
    *   Noted multiple albums can share the same `artist_id` (same artist, multiple albums)
*   One-to-many limitation:
    *   A simple foreign key from albums to artists allows one artist → many albums
    *   Does not represent many artists → many albums (multi-artist collaborations)
*   Many-to-many modeling via a join table:
    *   Introduced canonical pattern: add an intermediate relation (“join table”)
    *   Example domain: “problems” and “problem sets”
        *   One problem set has many problems
        *   A problem can appear in multiple problem sets (e.g., review sets)
        *   Join table holds `problem_id` and `problem_set_id`
    *   Relationship attributes can live in join table (e.g., points/weight of a problem *within* a problem set)
    *   Another example: students and courses
        *   Join table (e.g., “takes”) can store the grade

### SQL Joins: Inner Joins, Join Conditions, Cartesian Product, and Query “Lifecycle”

*   Returned to SQL practice via SQLBolt (`sqlbolt.com`), now introducing joins
*   Normalization context:
    *   Multiple “orthogonal” tables arise from normalization
    *   Normalization reduces redundancy and lets tables grow more independently
*   Join concept:
    *   A query involving multiple tables uses a join to combine relations
    *   Template discussed: `SELECT ... FROM tableA JOIN tableB ON ...`
*   Relational algebra intuition:
    *   Without a join condition, joining two tables produces the Cartesian product (cross product)
    *   Usually you add an `ON` condition to restrict pairings (e.g., match foreign key to primary key)
*   Inner join behavior:
    *   Returns only rows where the join condition matches
    *   Rows with no match are omitted (e.g., artist with no albums won’t appear)
    *   One-to-many produces multiple result rows for a single entity (artist appears once per album)
*   Continued availability of `WHERE`, `ORDER BY`, `LIMIT` after joining
*   Proposed mental model: queries often expand then narrow
    *   Joins tend to expand the result set (more columns, often more rows)
    *   `WHERE` and selection narrow to the data you actually need
*   Declarative nature of SQL:
    *   You describe the data you want, not how to navigate the tables
    *   DBMS chooses execution strategy, usually effective

### Outer Joins: Motivation and Variants (Left/Right/Full)

*   Motivated outer joins with artists/albums scenario:
    *   Want to include artists even if they have no albums in the database
    *   Outer join fills missing side’s columns with `NULL`
*   Data-cleaning example:
    *   To delete artists with no albums, you need to detect missing matches
    *   One approach: outer join then filter rows where album side is `NULL` (noted there are multiple ways)
*   Defined types:
    *   Left outer join: keep all rows from left table, match right when possible, else `NULL`s
    *   Right outer join: symmetric (keep all rows from right table)
    *   Full outer join: keep all rows from both sides, using `NULL`s where unmatched
*   Noted syntax convention:
    *   `JOIN` alone defaults to inner join
    *   If you write `LEFT`, `RIGHT`, or `FULL`, it is implicitly an outer join (no “left inner join”)

### Outer Join Practice: Buildings and Employees (SQLBolt)

*   Task: “find a list of all buildings that have employees”
    *   Started with `buildings` joined to `employees`
    *   Noted need for join condition matching building identifiers (e.g., `building_name = building`)
    *   Observed duplicates in results (one row per matching employee)
    *   Used `DISTINCT` to get unique building names
*   Task: “find a list of all buildings and capacity”
    *   Realized join unnecessary; simple `SELECT ... FROM buildings` (possibly `SELECT * FROM buildings`)
*   Task: “list all buildings and the distinct employee roles in each building, including empty buildings”
    *   Chose `LEFT OUTER JOIN` from `buildings` to `employees` to include empty buildings
    *   Initially omitted `ON` and saw Cartesian product; then added `ON building_name = building`
    *   Observed `NULL` roles for empty buildings (outer join behavior)
    *   Used `DISTINCT` over `(building_name, role)` pairs
    *   Clarified `DISTINCT` applies to the entire selected tuple, not just one column
