### From Cartesian Product to Joins (Conceptual Model)

*   Revisited the Cartesian product as the conceptual starting point
    *   Cartesian product pairs every row of one table with every row of another, causing a combinatorial ‚Äúexplosion‚Äù in size
    *   In relational algebra (as math), intermediate blow-up is fine because you can filter afterward
    *   In SQL practice, we usually want the filtered version directly: an inner join as a subset of those pairings
*   Illustrated the problem with a raw cross product example (instructor paired with courses they do not teach)
    *   Example intuition: an instructor like ‚ÄúEinstein‚Äù paired with unrelated courses (e.g., `CS11`) in the cross product
    *   Joins eliminate those nonsensical pairings by enforcing matching conditions

### Joins as ‚ÄúNavigating the Schema Graph‚Äù

*   Proposed an alternate mental model for joins: walking the schema diagram (graph) via relationships/foreign keys
*   Example relationship goal: ‚Äúevery instructor a student has taken a class from‚Äù (or the inverse)
    *   Requires navigating across several tables (a multi-table path)
    *   Even if you do not need the intermediate columns (e.g., building), you must traverse the tables to connect entities correctly
*   Emphasized that join conditions typically follow foreign key links
    *   Conceptually: cross product plus filtering
    *   Practically: write a `JOIN` with a condition that ties the foreign key to the referenced key

### Inner Join Variants: `NATURAL JOIN`, `USING`, and `ON`

*   Reviewed three common inner join syntaxes and how they differ
*   `NATURAL JOIN`
    *   Automatically joins on all columns with the same name in both relations
    *   Convenient when it matches intent, but ‚Äútreacherous‚Äù in practice
        *   Schema changes (e.g., adding a new same-named column) can silently break query meaning
        *   Example hazard: both `student` and `instructor` having a `name` column could cause incorrect joining on `name`
*   `USING (col1, col2, ...)`
    *   Like `NATURAL JOIN`, but you explicitly list the shared column names to join on
    *   Produces only one copy of the joined-on columns (merged duplicate columns in the output)
    *   Still requires the column names to match in both tables
*   `ON <predicate>`
    *   Most explicit and flexible: you write the join condition directly (commonly equality of key/foreign key columns)
    *   Columns do not need the same name because the predicate is explicit
    *   Predicate can be more complex than equality (full SQL expression), though equality on keys is the usual case
*   Noted a pragmatic preference: `ON` is verbose/ugly but commonly used because it is explicit and robust

### Outer Joins: Motivation and Usage

*   Defined what outer joins add beyond inner joins: include rows that do not match
*   `LEFT OUTER JOIN`
    *   Guarantees every row from the left table appears at least once
    *   If there is no match on the right side, right-side columns are filled with `NULL`
    *   Example described: `department LEFT OUTER JOIN instructor` (on department name) to include departments even with no instructors
*   Common use case: finding ‚Äúmissing‚Äù matches (often for cleanup/deletion workflows)
    *   To find empty/unreferenced entities, outer join then filter for `NULL` on the right side
    *   Example scenario described: deleting sections with zero enrollment by left-joining `section` to `takes` and filtering unmatched ones
*   Mentioned `RIGHT OUTER JOIN` and `FULL OUTER JOIN`
    *   `RIGHT OUTER JOIN` is symmetric to left join but guarantees coverage of the right table
    *   `FULL OUTER JOIN` includes all rows from both sides, with unmatched rows ‚Äútacked on‚Äù from either side

### Does Join Order Matter? (Inner Join vs Outer Join)

*   Asked whether there is a difference between starting a multi-table join chain from different tables (e.g., starting from `student` vs `instructor` vs `section`)
*   Clarified that for pure inner joins (with standard join conditions), the result set is the same regardless of order (conceptually)
*   Noted practical differences:
    *   `SELECT *` would yield different column ordering depending on join order (and is a reason to avoid `SELECT *`)
    *   For outer joins, order does matter because left/right preservation changes which unmatched rows appear

### ‚ÄúA Join Is with Everything So Far‚Äù (Incremental Relation Construction)

*   Reinforced the relational-algebra-inspired view: SQL operations always produce a relation
    *   Each `JOIN` takes the entire intermediate result ‚Äúso far‚Äù and joins it with one additional table
    *   This can be misunderstood as joining pairwise links, but it‚Äôs actually growing one intermediate ‚Äúblob‚Äù
*   Highlighted a trap: trying to rely on `NATURAL JOIN` mid-query because ‚Äúat this point only one column overlaps‚Äù
    *   The intermediate result has accumulated many columns; overlap is not just between the last two base tables

### Set Operations in Relational Algebra and SQL: `UNION`, `UNION ALL`

*   Introduced union as a frequently forgotten operation in SQL discussions
*   Compared relational algebra vs SQL on duplicates
    *   Relational algebra uses sets: duplicates are automatically eliminated
    *   SQL uses multisets (‚Äúbags‚Äù) by default: duplicates remain unless you use `DISTINCT`
*   `UNION`
    *   In SQL, `UNION` collapses duplicates (produces a set-like result)
    *   Even if input subqueries contain duplicates, the output is deduplicated
*   `UNION ALL`
    *   Preserves duplicates (bag semantics)
*   Worked through an example query goal: course IDs offered in Fall 2017 or Spring 2018
    *   In relational algebra terms: select matching rows from `section`, project `course_id`, then union the two sets
    *   In SQL terms: two `SELECT course_id FROM section WHERE ...` queries combined with `UNION`
*   Noted an alternative: a single query using `OR` in the `WHERE` clause could express the same logic

### Requirements and Naming Behavior for Set Operations

*   Clarified schema compatibility requirements for `UNION`-style operations
    *   The participating queries must return the same number of columns in the same order
    *   Types must be compatible
    *   Column names do not need to match between the two sides
*   Observed naming behavior: the output typically takes its column names from the first query
*   Demonstrated/mentioned column renaming in `SELECT` (e.g., `course_id` aliased to something like `cid`) and that the union still works

### `INTERSECT` and `EXCEPT` (Set Intersection and Set Difference)

*   `INTERSECT`
    *   Returns rows appearing in both inputs
    *   In the example (Fall 2017 vs Spring 2018 offerings), only a single common course appeared
*   Set difference in relational algebra corresponds to SQL `EXCEPT`
    *   Returns rows in the first query but not in the second
*   Noted informally that `EXCEPT` is used relatively rarely (the instructor had to recall the operator name)

### Rename Operator and Aliasing in SQL

*   Mentioned the relational algebra rename operator and mapped it to SQL aliasing
*   Column aliasing
    *   Use `AS` optionally (e.g., `course_id AS cid`) or omit `AS`
*   Table aliasing

### Equivalent Queries: Flat vs Nested Forms

*   Gave examples where different SQL formulations produce identical results

*   Expressed a preference (especially while learning) for flatter queries
    *   Nested queries can reflect a ‚Äúprocedural programming‚Äù mindset (‚Äúdo this, then use that‚Äù)
    *   Flatter forms better match the relational/declarative intent when possible
*   Mentioned joins that can be written ‚Äúin either direction,‚Äù reinforcing that many queries have multiple equivalent expressions

### Brief History and Standardization of SQL

*   Recounted origin: ‚ÄúSEQUEL‚Äù developed in IBM‚Äôs System R project, later renamed SQL (Structured Query Language)

*   Discussed standards process

    *   Standards published through bodies like `ANSI` and `ISO`

### Dialects and Lock-In: Database and Cloud Provider Analogy

*   Framed a strategic tradeoff: use vendor-specific features vs stick to portable ‚Äúgeneric SQL‚Äù

*   Drew an analogy to cloud platforms (e.g., embracing many managed services vs using only generic VMs/storage)

### SQL Data Types (Standard Types and Practical Reality)

*   Reviewed common standard SQL types
    *   `CHAR(n)`: fixed-length strings, padded with spaces (example use: fixed-format codes)
    *   `VARCHAR(n)`: variable-length strings up to `n`
    *   Mentioned modern practice: many systems (including SQLite) commonly use `TEXT` for strings without fixed length limits
*   Integer types: `INT` / `INTEGER` / `SMALLINT`
    *   Mentioned size details can be machine- or system-dependent, and aliases vary by database
*   Numeric types
    *   `NUMERIC(p,s)` / fixed-point decimal intended for exact decimal representation (e.g., money)
    *   Floating types: `REAL` (often 32-bit) and `DOUBLE PRECISION` (often 64-bit)
    *   Noted personal habit of using `REAL` but suggested `DOUBLE PRECISION` might be more appropriate in some cases
*   Reiterated that types diverge across SQL implementations; once you choose a database you should learn its recommended types/dialect

### Creating Tables and Integrity Constraints (Primary/Foreign Keys, `NOT NULL`)

*   Described the overall structure of `CREATE TABLE`
    *   Table name, list of columns, each with a type and optional constraints
*   Integrity constraints in `CREATE TABLE`
    *   `NOT NULL`: by default, columns are nullable unless specified otherwise
    *   `PRIMARY KEY`: can be specified inline for a single column, or separately (especially for composite primary keys)
    *   `FOREIGN KEY (...) REFERENCES other_table (...)`
        *   Foreign key list indicates referencing columns; referenced table (and optionally referenced columns) specified after `REFERENCES`
        *   If names match, referenced column list can sometimes be omitted; otherwise specify explicitly
*   Question addressed: must the referenced table exist before creating the table with the foreign key?
    *   In SQLite, creation order is flexible; enforcement occurs at `INSERT` time
    *   Noted circular references exist, and some systems allow temporarily suspending enforcement for bulk loading

### Data Modification Queries: `INSERT`, `DELETE`, `UPDATE`, and Upsert

*   `INSERT`
    *   Form described: `INSERT INTO <table> VALUES (...)`
    *   Recommended listing column names explicitly in `INSERT INTO <table>(...) VALUES (...)` to avoid depending on implicit column order
    *   Listing columns also enables omitting some fields (auto-generated keys, defaults)
    *   Mentioned default values: modern systems can fill omitted columns with defaults if configured
*   `DELETE`
    *   Form described: `DELETE FROM <table> WHERE ...` to target specific rows
*   `UPDATE`
    *   Form described: `UPDATE <table> SET col=new_value, ... WHERE ...`
*   Upsert (non-standard but common extension)
    *   Insert if key absent; otherwise update existing row
    *   Mentioned it avoids multi-step logic, but syntax varies and can be ‚Äúugly‚Äù

### More `SELECT` Details: Expressions, `DISTINCT`, and Avoiding `SELECT *`

*   Reviewed `DISTINCT` vs default `ALL`

*   Noted `SELECT *` is useful for quick testing but discouraged in durable queries

*   Mentioned that `SELECT` items can be arbitrary expressions, not only column names

### String Concatenation and Computed/Generated Columns (SQLite Example)

*   Demonstrated/outlined a real schema design from a scheduling system (referred to as ‚ÄúMarmot‚Äù)
    *   Goal: avoid redundant storage by deriving components (e.g., course prefix and course number) from a single stored string
*   Discussed generated columns (SQLite-specific syntax noted, but concept exists elsewhere)

### Cascading Actions: `ON DELETE CASCADE` and `ON UPDATE CASCADE`

*   Reinforced that foreign keys enforce referential integrity: referenced rows must exist; deleting referenced rows normally fails
*   Explained `ON DELETE CASCADE`
    *   If a referenced row (e.g., a course) is deleted, referencing rows (e.g., sections) are automatically deleted
*   Explained `ON UPDATE CASCADE`
    *   If a referenced key value changes, dependent references update automatically
    *   Noted this is most relevant when keys are ‚Äúnatural‚Äù values that might change (less common with surrogate integer keys)

### Complex String Validation Example: Timeslot Format Checks in the Database

*   Presented a ‚Äúgnarly‚Äù but illustrative example of enforcing a timeslot string format via SQL expressions and constraints
    *   Timeslot encoded as a string combining days + start time + duration (with a `+` delimiter)
    *   Desired constraints:
        *   Days are valid and appear in order (e.g., `M`, `T`, `W`, `R`, `F`, ...), with no repeats/out-of-order
        *   Time is valid and minutes are multiples of 5
        *   Prevent non-canonical equivalents (e.g., disallow something like ‚Äú13:75‚Äù as an alternative representation)
*   Described the approach:
    *   Use string search to find the `+` separator, then extract substrings for days, start time, and duration
    *   Validate the day sequence by repeatedly ‚Äúchomping‚Äù expected day letters in order using `REPLACE`, starting from a sentinel prefix
        *   Goal: after removing valid day markers in order, only the sentinel remains
    *   Validate time numerically by casting to a number and using modulo arithmetic to isolate minutes
    *   Reconstruct a canonical representation (including leading zeros) and compare back to the original to ensure exact formatting
