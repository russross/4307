## Tutorial: Data Stream Mining (Sampling, Bloom Filters, and Distinct Counting)

*   Motivated stream processing by modern high-volume data sources (search, social media, finance, IoT) where storing everything is infeasible.
*   Contrasted traditional DBMS “store then query” with streaming “process now or lose it,” emphasizing single-pass, incremental, approximate algorithms.
*   Explained two stream sampling approaches—element-wise sampling vs. key-/hash-based sampling—and why the wrong choice can severely bias analytics.
*   Introduced Bloom filters for space-efficient membership testing with *one-sided error* (no false negatives, possible false positives), including practical use in spam/whitelist filtering.
*   Presented the Flajolet–Martin family of algorithms for estimating distinct counts via trailing zeros in hashes, and the need to reduce variance using multiple hashes/groups/median; DGIM was listed but not covered.

### Why Data Streams Matter (Scale and Examples)

*   Illustrated the volume and velocity of modern data:
    *   Search engines processing very large numbers of queries per second.
    *   Social platforms (e.g., Twitter) producing large volumes of posts continuously.
    *   Financial markets generating massive numbers of daily transactions.
    *   IoT sensors producing continuous streams (smart home temperature readings, GPS from vehicles, etc.).
*   Highlighted the core challenge: data arrives too fast and in too great a volume to store and repeatedly scan in the traditional way.

### Traditional Database Processing vs. Stream Processing

*   Traditional DBMS (“comfortable model”):
    *   Data arrives and is stored first; queries happen later (hours/days/months later).
    *   Multiple passes over data, random access to records, and exact answers are expected (especially in relational systems).
    *   Works well when data fits in storage/memory and time allows.
*   Stream processing (“process now or lose it”):
    *   Data flows past; you often cannot “go back upstream” to retrieve what you missed.
    *   Must process incrementally, updating a small state per arriving element.
    *   Typically only a single pass over the data is possible.
    *   Approximate answers are common and necessary, not a failure.
*   Concrete motivating examples:
    *   Network routers seeing millions of packets per second (can’t store everything; missed packets can’t be recovered).
    *   Satellites transmitting terabytes of imagery per day.
    *   Social media platforms receiving extremely high post volume.
*   Emphasized that approximate results are often sufficient for decision-making (e.g., “~2.3M unique visitors ±2%”).

### Sampling from Data Streams

*   Motivated sampling as a “first line of defense” when data volume exceeds processing capacity.
*   Stated that stream sampling is subtle: it’s not only “randomly selecting elements,” but selecting them in a way that preserves the properties the analysis depends on.
*   Introduced two approaches:
    *   *Element-wise (naïve) sampling*: each arriving element is included independently with probability `p` (e.g., 10%).
    *   *Key-based / hash-based sampling*: inclusion is determined by hashing a key attribute (e.g., `user_id`) so that all records for a sampled key are included and all records for a non-sampled key are excluded.
*   Noted that choosing the wrong sampling method can yield misleading analytics even when the code is correct.

### Sampling Scenario: Query Stream Analytics (Why Naïve Sampling Fails)

*   Set up a concrete stream record format: `user_id`, `query_text`, `timestamp`.
*   Goal: analyze user behavior (queries per user, repeated searches, behavior over time) under limited processing capacity (e.g., needing a 10% sample).
*   Problem with element-wise sampling:
    *   Example user submits 10 queries.
    *   With 10% element sampling, expectation is ~1 query retained.
    *   This breaks per-user statistics:
        *   Estimating “average queries per user” from the sample can collapse toward ~1 instead of the true 10.
    *   Repeated-query detection becomes extremely unlikely:
        *   If a user repeats a query twice, probability both occurrences appear in a 10% element-sample is about `0.1 * 0.1 = 0.01` (1%).
        *   This makes it look like “almost nobody repeats,” even if many users do in the full data.
*   Presented the fix: key-based sampling by `user_id`.
    *   Hash each `user_id` into a bucket range (example used `0` to `9` via mod).
    *   If hash result meets a criterion (e.g., equals `0`), include *all* of that user’s queries; otherwise include none.
    *   Key property: the same key hashes consistently, so related records stay together.
*   Takeaway: when analysis depends on relationships among elements sharing a key, use hash-/key-based sampling rather than independent element sampling.

### Bloom Filters: Motivation and One-Sided Error

*   Introduced Bloom filters (Barton Howard Bloom, 1970) as a widely used, space-efficient data structure for membership queries: “Is element `x` in set `S`?”
*   Contrasted with exact structures:
    *   Hash tables / balanced trees provide exact membership but can be too memory-heavy at very large scale (e.g., billions of items).
*   Core Bloom filter property: *one-sided error*:
    *   If the Bloom filter says “not present,” it is *definitely not present* (no false negatives).
    *   If it says “present,” it is *probably present* (false positives are possible).
*   Explained why the asymmetry is useful in many domains:
    *   False positives are inconvenient (extra checks), while false negatives can be catastrophic (rejecting legitimate items).
*   Brief class discussion tied to domains like medical screening as an analogy: tolerating false positives more than false negatives depending on context.

### Bloom Filter Application Example: Email Whitelist / Spam Filtering

*   Scenario: large email provider maintains ~`10^9` known-good sender addresses (a whitelist).
*   Incoming email decision:
    *   If sender is on the good list, deliver directly.
    *   If not, run expensive spam analysis (content scanning, ML classifier, reputation checks).
*   Resource problem with exact storage:
    *   `10^9` addresses × ~20 bytes/address ⇒ ~20 GB raw, and ~30–40 GB with overhead (hash tables, pointers, load factor, etc.).
    *   Disk-based lookup would be too slow at high query rates (I/O bottleneck).
*   Bloom filter solution:
    *   Store the whitelist in a Bloom filter (example claim: ~1 GB, ~8 bits/address) with ~1% false positive rate.
    *   Enables RAM-resident, fast membership checks at very high throughput.
*   Interpreted 1% false positive rate:
    *   About 1% of unknown senders might be mistakenly treated as “known-good” and bypass expensive checks.
    *   This trade-off was presented as manageable compared to the massive performance gain.

### Bloom Filter Internals: Data Structure and Operations

*   Identified the three main components:
    *   A bit array of size `n` (positions `0` to `n-1`), initially all zeros.
    *   `k` independent hash functions mapping an element to indices in `0..n-1` (uniformly distributed, no bias).
    *   The conceptual set of elements (not stored directly).
*   Insertion procedure:
    *   For an element `x`, compute `h_1(x), ..., h_k(x)`.
    *   Set the corresponding `k` bit positions to `1`.
*   Query procedure:
    *   Compute the same `k` hash positions for `x`.
    *   If any of those bits is `0`, return “definitely not in the set.”
    *   If all are `1`, return “probably in the set.”
*   Clarified why “all ones” is not certainty:
    *   Bits can be set by other elements due to collisions; a queried element may “match” bits set by different insertions.

### Why False Positives Happen (and Why They’re Unavoidable)

*   Explained that bits do not “belong” to specific elements:
    *   Setting bit `42` doesn’t encode *which* element caused it—only that *some* inserted element hashed there.
*   False positive mechanism:
    *   A non-inserted element `x` hashes to `k` positions that happen to already be `1` from other elements.
    *   The filter incorrectly returns “probably present.”
*   Emphasized the key asymmetry:
    *   Inserted elements will always find their `k` positions set (Bloom filters don’t clear bits in the standard version), so false negatives do not occur.
*   Noted that false positive probability increases as the bit array becomes more filled with `1`s, motivating careful sizing.

### Small Worked Example: Collisions and a False Positive

*   Used an illustrative small Bloom filter with bits indexed `0..9` and `k = 3` hash functions.
*   Inserted element “cat”:
    *   Hash positions: `1`, `5`, `8` (bits set to `1`).
*   Inserted element “dog”:
    *   Hash positions: `0`, `5`, `6`.
    *   Observed collision at bit `5` (already `1`), showing overlaps are normal.
*   Queried:
    *   “cat” hashes to `1`, `5`, `8` ⇒ all `1` ⇒ correctly “present.”
    *   “dog” hashes to `0`, `5`, `6` ⇒ all `1` ⇒ correctly “present.”
    *   “rat” (not inserted) hashes to `1`, `5`, `6` ⇒ all `1` ⇒ incorrectly “present” (false positive).
*   Connected the example to the general rule: more insertions ⇒ more `1`s ⇒ higher collision rate ⇒ higher false positive rate.

### Query Processing Steps and “First-Pass Filter” Usage Pattern

*   Summarized membership query steps as a procedure:
    1.  Compute `k` hash values for `x`.
    2.  Check the corresponding `k` bits.
    3.  If any bit is `0`, return “definitely not in set.”
    4.  If all bits are `1`, return “probably in set.”
*   Re-explained the no-false-negative guarantee:
    *   An inserted element sets its `k` bits to `1`, and standard Bloom filters never clear them.
*   Highlighted the “first-pass filter” pattern:
    *   Use Bloom filter to avoid expensive operations (e.g., disk lookups).
    *   If Bloom filter says “no,” skip the expensive check.
    *   If Bloom filter says “maybe,” perform the exact (costly) verification.
    *   False positives translate into occasional unnecessary work, not incorrect “definitely not” answers.

### Bloom Filter False Positive Rate (Mathematical Sketch)

*   Introduced parameters:
    *   `n`: bit-array size.
    *   `m`: number of inserted elements.
    *   `k`: number of hash functions.
*   Used a “darts and targets” intuition:
    *   Each hash sets one of `n` positions; total “throws” ≈ `k*m`.
*   Gave the probability a specific bit remains `0` after insertions:
    *   Approximately `(1 - 1/n)^(k*m)` (with an exponential approximation using `e` for large `n`).
*   Indicated that this leads to an expression for false positive rate based on how many bits become `1`, but noted the class would skip deeper formula work due to time/complexity.

### Counting Distinct Elements: Flajolet–Martin Algorithm

*   Motivated the distinct counting problem on streams where exact tracking (e.g., a set of all unique items) is too memory-intensive.
*   Core observation:
    *   Hash each distinct element uniformly to a binary number.
    *   Probability patterns of trailing zeros:
        *   About half end in `0`.
        *   About a quarter end in `00`.
        *   About an eighth end in `000`, etc.
    *   Longer trailing-zero runs are rarer and become more likely to appear as the number of distinct elements grows.
*   Basic algorithm:
    *   For each stream element `x`, compute hash `h(x)` and count trailing zeros.
    *   Track `R = max` trailing-zero count observed so far.
    *   Estimate distinct count as `2^R`.
*   Memory benefit:
    *   Only need to store the integer `R` (constant space), independent of stream length.

### Flajolet–Martin Example and Variance Problem

*   Walked through a toy example with four names (e.g., Alice, Bob, Carol, Dave) and their hashed binary values.
*   Counted trailing zeros per hashed value (example tail lengths like `2`, `4`, `1`, `3`).
*   Observed `R = 4`, producing estimate `2^4 = 16` despite only 4 true distinct elements.
*   Explained that nothing “broke”—the method has inherent variance:
    *   An unusually long trailing-zero hash (an outlier) can inflate the estimate substantially.
    *   Single-hash estimates are restricted to powers of two, so they cannot match arbitrary true counts exactly.

### Reducing Variance: Multiple Hashes, Grouping, and Median-of-Averages

*   Proposed improvement: use many hash functions instead of one (e.g., dozens).
*   Noted a pitfall: simple averaging of `2^R` across hashes can be dominated by rare outliers (very large `R` values).
*   Described the combined approach:
    *   Use multiple hash functions and track `R_i` for each.
    *   Divide the hash functions into groups (example: 64 hashes split into 8 groups of 8).
    *   Within each group, compute an average of the estimates (enabling non-power-of-two intermediate results).
    *   Take the median of the group averages to reduce sensitivity to outliers.
*   Gave a small numeric illustration (multiple estimates per group, compute group averages, then take the median).
*   Mentioned that modern algorithms such as HyperLogLog refine this approach and can achieve ~1–2% error using only kilobytes of memory.

*   Noted time constraints and stated DGIM (sliding-window counting) would not be covered in this session.
*   Summarized main concepts to remember:
    *   *Stream data model*: data arrives too fast to store; algorithms are typically one-pass and incremental.
    *   *Hash-/key-based sampling*: preserves per-key relationships needed for many analytics tasks; avoids biases from element-wise sampling.
