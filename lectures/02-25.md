### Tables vs indexes: B+tree vs B-tree discussion

*   Instructor revisits the structural distinction:
    *   SQLite tables (rowid tables) are stored as B+trees
    *   SQLite secondary indexes are stored as B-trees (not B+trees)
*   Explanation for why B+trees are good for rowid tables:
    *   Intermediate nodes only store compact keys (row IDs are small integers, encoded compactly)
    *   Actual row payloads live in leaf nodes
    *   Copying row IDs through intermediate levels is cheap, allowing dense packing
*   Explanation for why indexes are not stored as B+trees in SQLite:
    *   Index keys can be large (e.g., text titles or multi-column keys)
    *   In a B+tree, keys would be duplicated at multiple levels, which is expensive for large keys
    *   The “value” in an index often includes a small rowid reference, so separating payload doesn’t save much space

### Decoding page 1 at the byte level: B-tree page header and cell pointer array

*   Page 1 specifics:
    *   The first `100` bytes of the file are the database header
    *   The B-tree page header for page 1 begins immediately after that at offset `100` (within page 1)
*   Identifying the page header fields and matching them to bytes:
    *   Page type byte indicates an interior table B-tree page (`0x05`)
    *   A 2-byte “first freeblock” pointer (often `0` when there are no freeblocks)
    *   A 2-byte “number of cells” field (expects `3` cells for the catalog root)
    *   A 2-byte “start of cell content area” pointer (where payloads begin near the end of the page)
    *   A 1-byte “fragmented free bytes” count
    *   For interior pages: a 4-byte “rightmost pointer” (instructor decodes a value that matches the earlier observed right link, around `2288`)
*   Cell pointer array interpretation:
    *   Immediately after the page header is an array of 2-byte offsets, one per cell
    *   Offsets typically descend because new cells are appended at the end of the page and grow backward
    *   The cell pointers provide the canonical ordering even if physical placement differs

### Varints (variable-length integers) in SQLite

*   Introduces SQLite’s `varint` encoding (transcribed as “variant” in places):
    *   1 to 9 bytes long
    *   High bit set (`1`) means “more bytes follow”
    *   The remaining 7 bits contribute to the numeric value
    *   Special case: the 9th byte uses all 8 bits so values can cover full 64-bit range
    *   Big-endian significance: earlier bytes are more significant
*   Why varints matter in SQLite:
    *   Row IDs often start at `1` and grow, so early values are small and encode compactly
    *   Length fields and other “usually small” values benefit from compact encodings
    *   Negative values generally require the full 9-byte encoding

*   Table leaf cell structure:

    *   Works through continuation bits and bit shifting
    *   Correctly recovers the rowid as `94721`
    *   Notes the trial-and-error involved in getting varint reconstruction correct
*   Record format within the payload:
    *   Each record has a record header and a record body
    *   Record header begins with a varint indicating the total header size in bytes (including the size varint itself)
    *   After that are “serial type” varints, one per column, describing:
        *   The type (null, integer width, float, text, blob)
        *   For text/blob, the length in bytes
    *   The body follows immediately, containing the raw bytes for each column in order

### Serial types and the “INTEGER PRIMARY KEY == rowid” special case

*   Instructor notes a key SQLite space-saving rule:
    *   When a table declares an `INTEGER PRIMARY KEY`, that column is an alias for the rowid
    *   The rowid is stored structurally (in the cell header), not as an ordinary column value
    *   To avoid duplication, SQLite stores `NULL` in the record body where that column would be
    *   Correct interpretation requires knowing the table schema
*   Applies this to the decoded record:
    *   First serial type is `0` (NULL), corresponding to the `INTEGER PRIMARY KEY` column (filled in from the rowid `94721`)
*   Notes on SQLite typing:
    *   Declared column types are largely advisory; SQLite uses dynamic typing/affinity
    *   Values are encoded based on the value stored rather than strictly the declared schema type
