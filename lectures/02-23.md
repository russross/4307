### SQLite Storage Model: Single File, Page-Oriented

*   SQLite stores everything “somewhat unusually” in a single file:
    *   Fits its goal as an embedded database library (not a server).
    *   It is not a distributed system; most databases discussed are client-server (newer ones may be more distributed).
*   SQLite can create a journal file or accessory files depending on mode, sometimes temporary.
*   The database file is page-oriented with a fixed page size.
    *   Historical default mentioned: `512` bytes.
    *   Current common default mentioned: `4096` bytes (4 KiB).
    *   Pages are numbered starting at `1` (not `0`).
*   Rationale for 4 KiB:
    *   Roughly matches common OS-exposed block sizes for storage (even if physical sectors differ).
    *   Database I/O often happens in 4 KiB units.
*   Larger page sizes:
    *   Load multiple disk blocks per database page.
    *   Can change performance characteristics and overhead.
*   High-level mental model:
    *   The database file is “a big bunch of blocks.”
    *   Each page is used for one of a few things; focus will be on B-tree pages.

### SQLite B-Tree Page Types and B-Tree vs B+ Tree Concepts

*   A SQLite B-tree page can be one of four variants:
    *   Table B-tree interior page
    *   Table B-tree leaf page
    *   Index B-tree interior page
    *   Index B-tree leaf page
*   Clarification: table B-trees are B+ trees in SQLite.
*   Review question: difference between B-tree and B+ tree:
    *   B-tree: “satellite information” (data/payload) can appear in internal nodes and leaves.
    *   B+ tree: payload appears only in leaves; internal nodes are indexing-only.
*   SQLite table B-tree indexing key:
    *   Always indexed by `rowid` (an integer unique per row), regardless of table schema.
    *   Declaring an `INTEGER PRIMARY KEY` makes it match the `rowid`; otherwise SQLite assigns a hidden rowid.
*   Why B+ trees help:
    *   Internal nodes replicate only the key (`rowid`), allowing many entries per page.
    *   Leads to wide fan-out and shallow trees.
    *   Core performance idea: disk/page retrieval is expensive; searching within an in-memory page is cheap.
*   Interior vs leaf pages:
    *   For table B+ trees:
        *   Interior pages contain child pointers and keys but not row payload.
        *   Leaf pages contain entire row contents (all columns) but no children.
    *   For index B-trees:
        *   Payload may be stored in interior or leaf nodes (difference emphasized as “B-tree, not B+ tree” behavior).
        *   Leaf pages have no children; interior pages have children.

### Overflow Pages and Why Databases Avoid Large Objects

*   SQLite limits how much of a row payload fits in a B-tree cell:
    *   A fixed maximum size is enforced (mentioned as roughly around a quarter-page, with complicated rules).
*   If a row/field is too large:
    *   The first portion is stored in the B-tree cell.
    *   The remainder goes to overflow pages.
    *   The cell points to the overflow chain/pages.
*   This pattern is common across database systems:
    *   Preserves the search characteristics of the tree (fan-out and predictable page usage).
    *   Still accommodates occasional large values.
*   Side note: databases are not optimized for large binary objects (e.g., JPEGs):
    *   Better to store large objects elsewhere and store links/IDs in the database.
    *   Databases are optimized for lookup, linking, and relational operations, not for interpreting large binary blobs.
*   For assignment simplicity:
    *   The class will work with data that does not trigger overflow behavior.

### SQLite Database Header (First 100 Bytes) and Magic Number

*   The first `100` bytes of the database file are a header containing file-level metadata.
*   The remainder of page 1 (after the 100-byte header) is an ordinary page, but with 100 fewer bytes available for its B-tree content.
*   Magic header string:
    *   First `16` bytes: `SQLite format 3\0` (a magic number / identifier).
    *   Common pattern: file formats include identifiers at the beginning for easy detection.

### Page Size Encoding and the 64 KiB Special Case

*   Page size is stored in `2` bytes and must be a power of two in a valid range:
    *   Between `512` and `32768`, or
    *   A stored value of `1` represents `65536` (64 KiB).
*   Explanation for the `1` sentinel:
    *   `65536` does not fit into an unsigned 16-bit integer (max `65535`), so SQLite uses `1` to represent 64 KiB.
*   Note on “K” units:
    *   In this lecture, “K” means `2^10` (1024), not 1000 (with the aside that hard drive marketing often uses metric).

### Returning to SQLite: B-Tree Page Structure (Common Across Databases)

*   Back to the structure of pages used for B-trees.
*   Page 1 special case:
    *   Has the 100-byte database header, then behaves like a normal B-tree page for the remainder.
*   General B-tree page layout components:
    *   Page header: `8` bytes for leaf pages, `12` bytes for interior pages.
    *   Cell pointer array.
    *   Unallocated/free space region.
    *   Cell content area (cells packed from the end of the page backward).
    *   Reserve region at the end (ignored in this class).
*   Explanation of “cells”:
    *   A cell corresponds to (roughly) one row entry (or a partial row when overflow is used).
*   Why cells are packed at the end:
    *   Cells are variable-length, so packing from the end allows growth/shrinkage.
    *   The pointer array at the top provides fast access to cell locations.
*   Two “orders” within a page:
    *   Pointer array order is the logical/key order.
    *   Cell content area order is physical/packed order.
*   Handling deletes/updates:
    *   Deletions may leave gaps; the system can defer compaction to avoid unnecessary processing.
    *   The header tracks free space and fragmentation (e.g., start of free blocks and fragmented free bytes).

### Page Splitting and Disk Ordering of B-Tree Pages

*   SQLite follows standard B-tree semantics:
    *   Fill a page as much as possible.
    *   When full, split into two roughly equal pages.
    *   Insert a separator entry into the parent.
    *   If there is no parent, create a new root.
*   Implication for on-disk page ordering:
    *   The root of a general B-tree is often not “early” in the file; it can appear later because it is created by later splits.

### Detailed B-Tree Page Header Fields (Including Rightmost Child Pointer)

*   The first byte of a B-tree page indicates page type:
    *   Interior index, interior table, leaf index, leaf table (with numeric codes like 2, 5, 10, 13 mentioned).
*   Header fields discussed:
    *   Offset to the first freeblock on the page.
    *   Number of cells on the page.
    *   Start of the cell content area.
    *   Number of fragmented free bytes.
*   Interior-page-only fields (extra 4 bytes, making `12` total):
    *   Page number of the rightmost child.
*   Explanation of rightmost child:
    *   For `n` keys in an interior node, there are `n+1` children.
    *   SQLite stores:
        *   A left-child pointer in each interior cell (associated with that key), and
        *   A single “rightmost child” pointer in the page header (not attached to a specific key).
