## Views, Transactions, Constraints, and Indexes (with CodeGrinder Examples)

*   Views as *virtual tables* for restricting access, simplifying queries, and presenting “pre-digested” (denormalized) data without changing the normalized core schema
*   Concrete CodeGrinder views to centralize complex authorization and “search across many fields” behaviors that would otherwise be repeated throughout application code
*   How views work in SQL (`CREATE VIEW ... AS ...`), including view composition, aggregate/summary views, and the distinction between virtual vs materialized views
*   Transactions and ACID semantics, especially isolation/serializability, with discussion of concurrency control approaches (SQLite locking vs MVCC; notes on distributed databases)
*   Integrity constraints (including foreign keys and `CHECK`) and a performance story showing how missing indexes can make constraint enforcement catastrophically slow; intro to B-trees/B+trees and why indexes matter

## Lecture Outline

### Opening and Motivation: Why Talk About Views?
*   Instructor transitions from prior material to “talk about views for a minute,” referencing that the class “saw a little bit of this last time.”
*   Recalls a previously-discussed schema (the *university* schema) and uses it as a baseline for thinking about what a user should be allowed to see.
*   Core motivation: sometimes you do *not* want to show the entire logical data model to every user.
    *   Multi-user databases often have security models giving restricted access to some users.
    *   Example: a campus “Banner” database where groups need access to some student tables, but not all attributes (e.g., some sensitive fields).
*   Emphasizes this is not only about granting access to whole tables; access may need to be more refined (subset of columns/rows, joined data, etc.).
*   Introduces views as a way to present only the information you want accessible, and then grant permissions to the view rather than the underlying tables.

### Views as Convenience and “Pre-digested” Data
*   Second motivation for views: simplify application logic by presenting combined/processed data in a convenient form.
*   References prior discussion of a scheduling system as an “extreme version,” but shifts to a simpler, more concrete example.
*   Clarifies terminology:
    *   A “virtual table” is a view.
    *   A view does not store data (in SQLite’s supported form); it stores the query definition and runs it on demand.
*   Notes that some DBMSs also support *materialized views*, which physically store the view result for performance (to be revisited later).

### CodeGrinder Domain Overview: Two “Sides” of the Data
*   Introduces CodeGrinder as the main example of where views are useful.
*   Describes the assignment/problem data model at a high level as having two sides:
    *   Problem-definition side:
        *   Problem types (with actions).
        *   Problems, each consisting of a sequence of steps.
        *   Steps can vary by problem type across steps.
        *   Problem sets as collections of problems.
    *   User/assignment side:
        *   Assignments reference user, course, and problem set.
        *   Commits: each submission (roughly once per step) creates associated commit data.
*   States a common need: start with a user and figure out which problem sets they should be able to access.

### CodeGrinder View Example: `user_problem_sets` (Convenience + Authorization)
*   Presents a view that joins assignments to course and user data to avoid repeating joins in many places.
*   What the view provides (as described):
    *   Course name and unique course identifier (used for directory naming and URL “slugs”).
    *   Course “note” field (Canvas-facing name).
    *   User identity fields (username/email).
*   Explains this view arose because the instructor repeatedly needed the same join while “poking around” at a console, but it also simplifies application logic.
*   Extends from convenience into security/authorization: determining which problem sets a user is allowed to download.
*   Describes two access cases, combined via a `UNION`:
    *   Student/user case: users can access problem sets they have been directly assigned (clicked through via Canvas).
        *   Start from assignments, join problem sets by `problem_set_id`.
        *   Mentions there had been a special-case path for a quiz feature that was later removed.
    *   Instructor/TA case: instructors can access every problem set assigned to any student *in courses where they are instructors*.
        *   Rationale: avoid the situation where someone is a TA in one course but a student in another and should not get broad access.
        *   Path described:
            1.  Start with users marked as instructors.
            2.  Join to assignments to determine which courses they are instructors for (the “pinning” to courses).
            3.  From those courses, join back to *all assignments in those courses* across all users.
            4.  Ultimately select the relevant problem set identifiers.
*   Notes an important modeling wrinkle:
    *   “Instructor” is logically a (person, course) property, but the system (due to campus data feeds) represents it in an unusual way (tied to assignment records).
*   Explains why centralizing this logic in a view helps security:
    *   Instead of “check permission then fetch,” many requests can be written as a join through the authorization view.
    *   If the join finds no row, the user was not authorized—authorization is enforced by query structure.

### More CodeGrinder Authorization Views (Problem- and User-Level)
*   Describes extending the authorization pattern:
    *   From problem sets down to individual problems (a view defining which problems a user can access).
    *   Another view for “which users can see which other users.”
        *   Rule described: instructors see students in their classes; a user can see themselves.
*   Connects to operational tooling: a `grind student` command.
    *   Pulls the latest submission for a student.
    *   Creates a temporary directory and launches a subshell to inspect/run code.
    *   On shell exit (e.g., Ctrl-D), cleans up the temporary directory.
    *   Running grading inside the shell can execute tests without storing results.
*   Notes this command relies on authorization logic so that an instructor can access submissions for students in their own classes, but not for unrelated classes where they are merely a student.

### Views as “Denormalization Without Corrupting the Core Schema”
*   States a general philosophy:
    *   Relational design stores data in a *natural/normalized* representation, not optimized for a particular access pattern.
    *   Application needs often want bundled fields “likely to be needed at the same time,” which pushes toward denormalization.
*   Views provide a compromise:
    *   Keep stored data normalized and well-modeled.
    *   Provide denormalized, convenient projections for usage, without changing the underlying storage model.
*   Notes that after adopting this pattern, the instructor “went a little nuts with it,” adding many views.

### CodeGrinder Search Views: `assignment_search_fields`
*   Problem: when downloading an assignment via tooling, the instructor doesn’t know numeric assignment IDs and wants “search terms” instead.
*   Defines a view that maps:
    *   `assignment_id`
    *   A single concatenated string of many searchable fields.
*   Describes the concatenation approach:
    *   Uses SQL string concatenation (SQLite `||` operator).
    *   Includes text such as:
        *   Assignment Canvas title
        *   Course name
        *   User’s name
        *   User’s email
        *   Problem set unique ID and “friendly name”
        *   Tags (included to make them searchable)
*   Benefit: searching becomes a trivial `LIKE` query across one combined field rather than multi-field logic in application code.

### CodeGrinder Search Views: Problem Set Search with Aggregation (`group_concat`)
*   Second search problem: searching for problem sets by keywords like “risk,” “arrays,” etc., including metadata for problems in the set.
*   Describes join path:
    *   Problem sets joined through a join table to problems, to bring in per-problem metadata.
*   Uses aggregation:
    *   Uses SQLite aggregation function `group_concat` to concatenate problem-level fields (unique IDs, notes, tags) into a single string per problem set.
    *   Groups by problem set, concatenating across all its problems.
*   Notes a correctness issue in the view definition:
    *   The view selects fields not included in the `GROUP BY`, relying on the fact they’re constant within the group.
    *   Instructor remarks it’s “wrong” in strict SQL terms and should be fixed.

### View Definition Basics and Simple Examples
*   General syntax:
    *   Use `CREATE VIEW`, provide a view name, then `AS` followed by a query expression.
    *   Any legal SQL query expression can define a view.
*   Once created, the view name can be referenced like an ordinary table for querying.
*   Emphasizes the “virtual” nature:
    *   The joins and computation happen on-the-fly (unlike populating a physical table).
    *   Contrasts with a prior demo where a table was created and populated (materialized-like behavior).
*   Simple access-control examples from the university schema:
    *   A view of faculty/instructors without salary (projection to hide a sensitive column).
    *   A view of instructors in a particular department (row restriction).
*   Other practical uses:
    *   Time-series slicing: a view representing “current year” records instead of creating a new table per year (calls yearly tables a bad design).
    *   Soft deletes: keep all records for compliance, add a “deleted” marker column, then create a view that filters to non-deleted records.
    *   Aggregated/anonymized reporting: provide department-level salary totals/averages without exposing individual salaries.
*   Notes that views can be defined using other views, and briefly mentions the idea (and oddity) of self-referential definitions.

### Materialized Views: Benefits and Maintenance Complexity
*   Reintroduces materialized views:
    *   Some DBMSs compute the view once and store the resulting data physically.
*   Highlights the maintenance challenge:
    *   Underlying base-table updates may not automatically update the materialized view.
    *   The view might involve complex aggregates (e.g., averages), making incremental maintenance nontrivial.
    *   Possible tools include triggers or scheduled rebuilds; often it’s a managed strategy rather than automatic.
*   Gives a motivating scenario:
    *   Snapshot “digested” data (e.g., recommendation features) updated periodically (e.g., once per day) where slightly stale results are acceptable.

### Updating Through Views: What Would It Mean?
*   Poses the question: what if you try `INSERT` into a view rather than into the underlying base table?
*   Notes ambiguity and failure modes:
    *   Missing required columns: a view that hides `salary` might prevent inserts if `salary` is required in the base table.
    *   The DBMS might reject the insert, or attempt to fill missing fields with `NULL` (if allowed).
*   Shows a more complex example idea:
    *   A view joining instructor info with department/building, then trying to insert through it raises questions:
        *   Which department should be chosen when only a building is provided?
        *   The view hides critical information needed to satisfy base-table constraints.
*   Discusses why updates-through-views can matter in real organizations:
    *   In large shared databases (e.g., campus Oracle systems with hundreds/thousands of tables), applications are “tenants” with restricted access.
    *   Those applications may need to insert/update, but cannot directly access all underlying tables.
    *   Reality: you often cannot just “change the schema,” because many unknown applications depend on it.
*   Mentions use of unique IDs like campus “D numbers” to avoid exposing sensitive identifiers like SSNs (a user-visible system change used to address privacy concerns).
*   Concludes this is often DBMS-dependent and organizationally constrained, not just a clean technical design exercise.
*   SQLite-specific note from discussion:
    *   SQLite does not allow updating views (so the entire class of complexity largely goes away there).

### Transition to Transactions: Definition and ACID
*   Shifts topic: “Nice transactions,” and formalizes what a transaction is.
*   Defines a transaction as a sequence of queries/updates treated as one unit of work.
*   Reviews ACID semantics:
    *   Atomicity: all-or-nothing execution.
    *   Consistency: constraints are preserved.
    *   Isolation: concurrent transactions behave as if serialized.
    *   Durability: once committed, changes persist.
*   Notes the default behavior in SQL systems:
    *   Each individual SQL statement implicitly runs as its own transaction unless an explicit transaction is started.
*   Explicit transactions:
    *   Start a transaction, run multiple statements, then end with either `COMMIT` (permanent) or `ROLLBACK` (undo all changes as if nothing happened).

### Isolation, Serializability, and Concurrency Control
*   Emphasizes isolation as a key value proposition:
    *   Multiple users run transactions concurrently.
    *   The database ensures results are consistent with some serial order (“serializability,” the gold standard).
*   Describes conflicts and blocking/aborts:
    *   A transaction may have to wait if it needs data modified by an uncommitted transaction.
    *   Systems may block, force one to wait, or abort one transaction depending on the implementation.
    *   A commit can fail due to serialization conflicts if the system cannot order transactions consistently with what was read/assumed.
*   Notes that the goal is maximum concurrency:
    *   Non-overlapping transactions should proceed concurrently.
    *   Lock granularity (row/page/column) is an implementation detail affecting concurrency.

### SQLite Locking vs MVCC (Postgres) and Other Strategies
*   Mentions SQLite’s basic approach (especially historically):
    *   Transactions begin effectively read-only, and the first write upgrades to a write transaction.
    *   Write transactions can lock out others, reducing concurrency compared to server DBMSs.
*   Contrasts with MVCC (Multi-Version Concurrency Control), common in Postgres:
    *   Writes create new versions of rows; old versions remain available temporarily.
    *   Read-only transactions can observe a consistent snapshot (as of a timestamp/transaction start) without blocking writers.
    *   The system can order commits to make concurrent execution appear serialized.
*   Notes there are many strategies with different trade-offs, and the details are engine-dependent.

### Distributed Databases and Global Transactions (High-Level)
*   Observes that “databases” and “distributed systems” are often treated as separate fields, even though many important systems are distributed databases.
*   Mentions hyperscaler-scale systems and goals:
    *   Example: Google Spanner (named for spanning the globe), used to support massive global applications.
*   Points out that transaction/concurrency control becomes even more interesting in distributed settings.

### Weaker Isolation Levels and Why Transactions Matter in Practice
*   Notes the SQL standard defines serializable as the strongest level but also weaker isolation levels.
    *   Example notion: allowing reads of uncommitted data (weaker guarantees) can improve concurrency/performance.
*   Returns to practical value:
    *   Transactions are a major value of DBMSs and solve hard concurrency problems for typical applications.

### Application Pattern: One Request = One Transaction
*   Describes a common web/app backend architecture:
    *   Client → server → database (three-tier architecture).
*   Recommended pattern:
    *   Handle a single request entirely inside a transaction.
    *   On success, `COMMIT`, then report success to the user.
    *   On failure (constraint violation, serialization failure, or business logic failure), `ROLLBACK` so nothing changes.
*   Argues this pattern “pushes” concurrency complexity into the database:
    *   Avoid server-side shared state; treat operations as database statements.
    *   Let the DBMS provide isolation and atomicity.

### Avoiding Premature “Infinite Scalability” Design (Jeff Dean Principle)
*   Discusses system design advice attributed to Jeff Dean:
    *   Don’t design for infinite scalability; it’s usually solving imaginary problems.
    *   Plan for growth (e.g., 10×), but beyond ~100× current scale you’re likely to get the wrong bottlenecks.
    *   Bottlenecks change as systems scale, so premature optimization misses real future constraints.
*   Applies the lesson to database choices:
    *   Starting with a relational database (e.g., Postgres) is usually the right move.
    *   Outgrowing modern databases is hard; if it happens, it’s a “good problem” and you’ll likely have resources to redesign.

### Integrity Constraints as Part of Consistency
*   Connects transactions to consistency via integrity constraints.
*   Notes databases can enforce business rules, not just structural rules.
    *   Example given: a checking account balance must stay above a threshold (enforced by constraints so violating transactions fail).
*   Reviews common constraint types:
    *   `NOT NULL`
    *   `PRIMARY KEY` and `UNIQUE` (uniqueness can apply to non-primary columns and to multi-column combinations)
    *   `CHECK` constraints using predicates supported by the DBMS
*   Notes the instructor’s earlier use of `CHECK` constraints for formatting/validation (e.g., time slot naming rules).

### Referential Integrity: Foreign Keys and Cascades
*   Highlights foreign key constraints as especially important:
    *   Prevent creating referencing rows with nonexistent referenced keys.
    *   Prevent deleting a referenced row if dependent rows exist, unless cascade rules are configured.
*   Discusses `ON DELETE` / cascading behaviors:
    *   Cascades can delete dependent rows automatically (e.g., deleting a user could delete their assignments and commits).
    *   Without cascade rules, foreign key constraints forbid the delete that would orphan references.
*   Notes constraints may be enforced even when doing so is computationally expensive—correctness comes first.

### Performance Story: `DELETE FROM commits` and the Need for Indexes
*   Shares a CodeGrinder incident:
    *   At semester end, attempted `DELETE FROM commits`.
    *   Expected it to take seconds, but it ran for a very long time (eventually canceled after a long wait).
*   Root cause analysis:
    *   The `commits` table had a self-referential foreign key (each commit can reference a parent commit).
    *   To enforce referential integrity while deleting a row, the DBMS must check whether any other row references it.
    *   Without an index on the parent-reference column, the DBMS must scan the entire `commits` table for each row being deleted.
    *   This turns a linear delete into an *O(n²)* pattern (full table scan per deleted row).
*   Fix:
    *   Create an index on the self-referential foreign key column.
    *   After indexing, rerunning `DELETE FROM commits` completes quickly because each “does anyone reference this?” check becomes a fast indexed lookup.
*   Takeaway:
    *   Foreign key enforcement often implies you need supporting indexes on referencing columns, especially for deletes/updates.

### `CHECK` Constraints with Subqueries (Power and Caution)
*   Notes that `CHECK` constraints can be surprisingly powerful, including subqueries.
    *   Example discussed: a `CHECK` that effectively enforces membership in a referenced table (similar to a foreign key).
*   Warns about complexity:
    *   Constraints might need re-evaluation when other tables change (because a subquery result can change even if the constrained row did not).
    *   Mentions an example idea where a constraint compares against an aggregate (e.g., average salary), which could imply widespread rechecking upon updates.

### Large Object Types: BLOBs and CLOBs
*   Introduces large object types:
    *   BLOB: Binary Large Object, arbitrary binary data with no DB interpretation.
    *   CLOB: Character Large Object, large text data (less commonly used explicitly now; many systems just use `TEXT`).
*   Design caution:
    *   Storing large binary assets (e.g., images) directly in database rows can harm performance (rows become huge).
    *   Often better to store large objects in an external object store (e.g., bucket storage) and keep only references/IDs in the database.
*   Connects this to upcoming discussion of on-disk formats and why typical database storage structures are not optimized for very large blobs in-row.

### Intro to Indexes and On-Disk Storage: B-trees and B+trees
*   Begins introducing indexes at a high level (with more detail promised later).
*   Notes that tables have an on-disk organization; common structures are B-trees / B+trees.
    *   Mentions balanced-tree properties (leaf nodes at the same depth).
    *   Briefly acknowledges nuances and that different systems use variants (B-tree vs B+tree).
*   Uses the term “stabbing query” for a tree lookup:
    *   Start at the root and follow the appropriate child pointers down to a leaf based on ordering.
*   Discusses what ordered trees enable:
    *   Efficient point lookup by the sorted key.
    *   Efficient range scans: find the start of a range, then scan forward/backward in order.
*   Contrasts with the failure mode:
    *   Queries on attributes that are not part of the sort order can require full table scans, which are inefficient.

### Primary Key Ordering vs Queries on Other Attributes
*   Assumes (with caveats) that tables are typically organized/sorted by primary key on disk.
*   Motivating example:
    *   A `movies` table with an integer primary key.
    *   Users want to look up by title, release year, budget, etc., not by integer ID.
*   Reiterates earlier design principle:
    *   Don’t denormalize/reorder storage purely based on anticipated queries.
    *   Instead, keep a “neutral” core storage model and use indexes to support efficient alternate access patterns.

### What an Index Is (Conceptually) and `CREATE INDEX`
*   Defines an index as essentially an internal auxiliary table:
    *   A partial duplicate containing the indexed columns plus a reference (typically the primary key) back to the base row.
    *   Stored in a structure optimized for lookup by the indexed key.
*   High-level syntax described: `CREATE INDEX` with a name, `ON` a table, and an attribute or set of attributes.
*   Explains how index lookup works:
    *   Use the index to find matching keys quickly.
    *   Retrieve the base table row via the primary key found in the index entry.
*   Connects back to the `commits` story:
    *   Indexing the self-referential parent key allows fast discovery of “children” referencing a commit being deleted.

### Index Design and Query Optimization (Preview)
*   Notes that adding/dropping indexes is a flexible tuning tool; indexes need not be fixed at initial schema creation.
*   Sets expectations for later in the semester:
    *   Study how queries are executed and how indexes are used.
    *   Practical optimization workflow: identify slow queries, determine if full scans or suboptimal access paths occur, then add appropriate indexes.

### Hash Indexes vs Tree Indexes (Why Trees Dominate in DBMSs)
*   Mentions that hashing can be used for indexing, but focuses on tree-based indexes in SQLite.
*   Discusses trade-offs:
    *   Hashing can be close to *O(1)* for exact-match lookups.
    *   But hash indexes are not versatile for range/prefix queries.
        *   Example: “find all movies that start with ‘F’” or “all movies released in 2017” benefits from ordered indexes.
*   Notes that in practice, tree index top levels are often cached, so performance can be near constant-time for many lookups while still supporting range scans.
