## SQLite Query Plans, Indexes, and Join Optimization

*   The lecture opened with a short discussion of current AI coding assistants, including changes to GitHub Copilot student access, the relative strengths of `Gemini`, `Claude Code`, and `Codex`, and why lightweight command-line agents can be more productive.
*   The main class topic was query implementation in SQLite, especially as preparation for the B-tree iterator assignment and an upcoming seven-part IMDb query project built around point lookups and range scans.
*   The instructor framed SQL as a classic leaky abstraction: to use databases well, programmers need “empathy for the machine” and an understanding of how storage structures, indexes, and execution actually work.
*   Using `EXPLAIN QUERY PLAN` and `EXPLAIN`, the lecture showed how SQLite executes a simple title lookup first as a full table scan and then, after adding an index on `primary_title`, as an efficient indexed search plus row lookup.
*   The lecture then moved into join optimization, comparing conceptual nested-loop joins with SQLite’s cost-based planning, automatic indexes, covering indexes, join ordering, and the effects of `WITHOUT ROWID`, concluding with course logistics about upcoming quizzes and the next assignment.

### SQL as a leaky abstraction and “empathy for the machine”

*   Before diving deeper into execution plans, the instructor paused for a conceptual aside.
*   He called SQL one of the canonical examples of a leaky abstraction, referencing Joel Spolsky’s “law of leaky abstractions.”
*   The point was that abstractions help manage complexity, but for nontrivial cases, programmers still need to understand what happens underneath.
*   In databases, that means understanding how the engine really executes a query and how data is physically stored.
*   He tied this to the idea of “empathy for the machine”:
    *   good engineering comes from understanding the physical limits and stresses of the system you are using
    *   software is no exception
*   To illustrate the idea, he used physical analogies:
    *   pulling a power cord from the wall by yanking the cable rather than the plug
    *   putting stress on the join between a cable and its connector
    *   carefully removing a `USB-C` cable by gripping near the laptop instead of torquing the cord
    *   seeing a device rest on its charging cable and immediately recognizing the strain it places on a weak point
*   He compared this to driving a car with or without a mental model of the machine and its physics.
*   The software version of that mindset is understanding the constraints of hardware and database storage structures.
*   One concrete example was storing `BLOB`s or images in SQLite:
    *   it is possible
    *   it is sometimes acceptable for small or infrequent cases
    *   but it is often not a good fit for the underlying B-tree structure
*   He explained that large objects introduce overhead because part of the row remains in the main B-tree while overflow storage is linked elsewhere.
*   The lesson was not that one must avoid such uses out of principle, but that knowing how SQLite works should influence design choices when performance and scale matter.

### Inspecting execution plans for a simple lookup

*   The instructor introduced SQLite’s built-in tools for inspecting execution:
    *   `EXPLAIN QUERY PLAN`
    *   `EXPLAIN`
*   He copied the IMDb database so he could safely modify indexes without affecting assignment inputs.
*   He removed the index on `titles(primary_title)` and reran the `Monsters, Inc.` lookup.
*   Although the query still felt fast interactively, he emphasized that this is misleading on a small test case.
*   `EXPLAIN QUERY PLAN` showed that SQLite had to `SCAN titles`.
*   He explained why a scan is necessary without an index:
    *   titles are not stored ordered by `primary_title`
    *   duplicate titles are possible
    *   therefore the engine cannot stop after the first match
*   He then used plain `EXPLAIN`, which exposes SQLite’s lower-level virtual machine program.
*   He described this as bytecode executed by an interpreter:
    *   opening tables for reading
    *   rewinding to the beginning
    *   reading columns
    *   comparing values
    *   outputting matching rows
*   He noted that the opcode listing is not pleasant to read, but it reveals that SQLite literally compiles a small program for the query.
*   This led to a broader point that many databases use a similar internal execution model.
*   He also mentioned that some high-performance modern systems go further and use JIT compilation, often with `LLVM`, to generate optimized native code for query fragments.

### Adding an index and comparing the new plan

*   The instructor recreated an index on `titles(primary_title)`.
*   Running `EXPLAIN QUERY PLAN` again produced a different result: SQLite now searched `titles` using the new index.
*   He connected this directly to the B-tree iterator assignment:
    *   the index is ordered by title
    *   the query can drill directly to the matching key range
    *   this is effectively a stabbing query followed by iteration over equal prefixes
*   Looking at the fuller `EXPLAIN` output, he pointed out that SQLite now opens both:
    *   the main table
    *   the index
*   The execution is more complex per match because it:
    *   seeks into the index
    *   finds matching entries
    *   uses the stored row identifier to fetch the full row from the table
*   Even so, it is far more efficient overall because the loop runs only over matching titles rather than every row in the table.
*   The iterator stops as soon as the prefix no longer matches, so very little work is wasted.

### Query optimization: from rule-based to cost-based planning

*   After showing that indexed and non-indexed plans differ, the instructor asked how the database decides which plan to use.
*   He contrasted two broad styles of optimizer:
    1.  rule-based optimizers
    2.  cost-based optimizers
*   Rule-based optimizers were described as collections of heuristics such as:
    *   if there is an index, use it
    *   if a certain join form appears, apply a specific transformation
*   He said older systems worked more like this, with lots of hand-coded decision logic.
*   Modern systems instead build candidate plans and estimate their costs.
*   The process he described was:
    1.  parse the SQL into a tree structure, much like compiling a programming language
    2.  generate alternative ways to execute each part
    3.  estimate the cost of each alternative
    4.  choose the cheapest plan under the internal cost model
*   Possible costs include things like:
    *   random disk reads versus sequential reads
    *   full scans versus indexed lookups
    *   B-tree search depth
    *   expected row counts
*   He noted that this planning happens during statement preparation, not during each row fetch.
*   Client APIs often expose this split as `prepare` versus `execute`, and prepared statements can be reused.
*   Because the full plan search space is huge, the optimizer relies on dynamic programming with memoization rather than brute force.
*   The instructor briefly reviewed dynamic programming as a way to solve a problem by combining remembered subproblem results.

### Join execution with `instructor` and `teaches`

*   The lecture then turned to joins using the university database.
*   He proposed a simple join to find what a specific instructor, `Katz`, teaches.
*   Conceptually, he reminded students that a join can always be thought of as nested loops:
    *   scan one table
    *   for each row, scan another table looking for matches
*   This direct implementation is undesirable because it tends toward `O(nm)` behavior.
*   Looking at the actual SQLite plan for `instructor NATURAL JOIN teaches`, the engine did something better:
    *   scan `teaches`
    *   search `instructor` using an automatically created index on the primary key
*   He interpreted this as taking each `teaches.id` and doing a stabbing query into `instructor`.
*   There was discussion of automatic indexes on primary keys, especially when the key is not just an alias for the rowid.
*   For the filtered version with `WHERE name = 'Katz'`, the plan changed again:
    *   scan `instructor`
    *   search `teaches` using a covering autoindex
*   He explained “covering index” to mean that the index itself contains all the needed columns, so SQLite does not need to follow the lookup back to the main table for that side of the join.

### Designing better indexes and using covering indexes

*   The remaining problem in the filtered join was that SQLite still had to scan the whole `instructor` table to find `name = 'Katz'`.
*   To fix that, he created an index on `instructor(name)`.
*   The new plan was better:
    *   search `instructor` using the `name` index
    *   search `teaches` using its covering index
*   He then asked whether the `instructor` side could also become covering.
*   He reduced the selected columns to just those actually needed from `instructor` and `teaches`, dropping columns like `dept_name` and `salary`.
*   Even then, SQLite still had to touch the main `instructor` table.
*   The reason was an important SQLite detail:
    *   the table’s logical primary key `id` was not an integer rowid alias
    *   an index on `name` contains `name` and a rowid reference, not the logical `id` value needed for the join
*   To solve this, he dropped the old index and created a composite index on `instructor(name, id)`.
*   With that index in place, the plan finally treated the `instructor` access as covering:
    *   search by `name`
    *   read `id` directly from the index
    *   avoid a second lookup into the table
*   He emphasized that this is not a dramatic asymptotic improvement, but it is a real constant-factor win.

### Multiway joins, join order, and `WITHOUT ROWID`

*   Next, the instructor expanded the example to a three-way join over `department`, `instructor`, and `teaches`.
*   The selected output included:
    *   department name
    *   department building
    *   instructor `id` and `name`
    *   course and section information from `teaches`
*   The main question was join order: where should execution begin?
*   He said the goal is to avoid repeated full scans and instead do at most one scan plus indexed lookups outward.
*   To make the storage model align better with intuition, he dumped the database to SQL using `.dump`, edited table definitions, and rebuilt the database with `WITHOUT ROWID` on eligible tables.
*   He described the purpose of this transformation as:
    *   making primary-key storage behavior more direct
    *   removing some of the rowid-versus-primary-key weirdness
    *   simplifying discussion of how lookups should work
*   When he asked SQLite for the three-way join plan, it chose to:
    *   scan `teaches`
    *   search `instructor` by primary key
    *   search `department` by primary key
*   He reasoned through why this might actually be better than starting from `department` or `instructor`:
    *   every `teaches` row must correspond to an `instructor`
    *   every `instructor` row must correspond to a `department`
    *   but not every `department` must have an instructor
    *   and not every instructor must currently have a `teaches` row
*   That means starting from `teaches` may avoid wasted work on rows that fail to produce output.
*   He suggested that a sufficiently refined cost model can infer this from schema constraints and use it during optimization.

### Filtering the multiway join and the role of statistics

*   Finally, he added a predicate such as `WHERE dept_name = 'Music'`.
*   This changed the plan so SQLite first searched `department` by its primary key.
*   After that, however, SQLite still performed a full scan of `instructor` to find matching department rows.
*   The instructor noted that this is the next obvious optimization target: add an index on `instructor(dept_name)` so the second step can also become a search rather than a scan.
*   He ran `ANALYZE` to compute statistics and see whether SQLite would revise its choice.
*   On this tiny dataset, the plan did not change.
*   He noted that SQLite’s statistics are fairly simple here, whereas other database systems often maintain richer statistics and may make more nuanced choices.
*   The lecture ended this technical section with the idea that schema design, indexing, and understanding the optimizer all work together.
