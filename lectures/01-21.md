## Relational Databases Foundations: Normalization, Keys, Joins, and a Relational Algebra Primer

*   Clarified how normalization in relational databases reduces redundancy and improves data integrity compared to flat files (e.g., consistent artist names via single referenced records).
*   Introduced core relational-model terminology: relations/tables, tuples/rows, attributes/columns, domains, atomic values, and the special role of `NULL`.
*   Defined database schema concepts and key types (superkey, candidate key, primary key) plus foreign keys and how they enforce relationships across tables.
*   Walked through a university-schema diagram to explain composite primary keys, join/bridge tables for many-to-many relationships, and how foreign keys connect multi-attribute keys.
*   Presented relational algebra operators and connected them to SQL query construction, emphasizing joins (including `ON` vs `USING` vs `NATURAL JOIN`) and a practical “expand → filter → aggregate → select/present” query-building strategy.

### Flat Files vs. Relational Databases: Data Integrity and Normalization

*   Student concern: issues like inconsistent spelling (e.g., artist/band name) seem like they could happen in a relational database too, depending on data entry.
*   Instructor clarification using the “albums and artists” example:
    *   In a flat file (single CSV), each album row repeats the band/artist name; redundancy makes inconsistent spelling likely across rows.
    *   In a normalized relational design:
        *   The artist/band exists once (single record).
        *   Albums reference that record via a relationship (foreign key), removing redundant repeated names.
        *   Misspelling could still occur, but it would be *consistent* because there is only one stored value.
*   Emphasis: simply storing a flat-file-like structure inside a database does not automatically provide integrity benefits; the benefits come from *normalizing* and using relationships to eliminate redundancy.

### Terminology Clarification: “Normalization” (Databases vs. Math/Statistics)

*   Student asks whether normalization here means statistical normalization (e.g., z-scores, min-max scaling).
*   Instructor: database normalization is different; it refers to *normal forms* in relational database design.
*   Brief discussion that “normalization” is a broader term used in multiple domains (linear algebra, statistics), but the database meaning is specific to schema design and redundancy reduction.

### Relational Model Terminology: Relations, Tuples, Attributes, Domains

*   Introduces a sample `instructor` relation/table with attributes like `ID`, `name`, `dept_name`, `salary`.
*   Terminology mapping:
    *   Relation = table
    *   Tuple = row
    *   Attribute = column
    *   Domain = set/type of allowed values for an attribute (e.g., integers, strings)
*   Atomic/scalar attribute requirement in the relational model:
    *   Proper relational design expects each attribute value to be atomic (single value).
    *   You “can’t store a list” in one column in a normalized relational model.

### Practical Databases vs. Pure Relational Model (Lists, JSON, Extensions)

*   Student notes that “every database lets you” store non-atomic values.
*   Instructor response:
    *   Many real systems relax strict relational rules for practicality/performance and simplicity.
    *   Common modern approach: store JSON strings/JSON values; some systems have native list/array types.
    *   This is considered an *extension* beyond the classic relational model and comes with tradeoffs.
*   Tie-in to normalization: *First Normal Form (1NF)* eliminates lists/repeating groups in columns.

### `NULL` as a Special Value and Its Complications

*   `NULL` is described as a special value member of every domain unless disallowed (e.g., via `NOT NULL`).
*   `NULL` is not equivalent to `0` and requires special comparison/logic rules.
*   Instructor notes that `NULL` causes complications in defining operations.

### Unordered Nature of Relations (Conceptual vs. Physical Storage)

*   Relations are fundamentally unordered conceptually:
    *   The DBMS is free to store rows in any order.
    *   Physical disk order exists but is not part of the relational model.
    *   Sometimes systems exploit storage order, but logically it’s irrelevant.

### Database Schema: Logical Structure and Creation

*   Defines database schema as the *logical structure* of the database.
*   Schema can be represented:
    *   As diagrams (ER/relational schema diagrams)
    *   As DDL statements (e.g., SQL `CREATE TABLE ...`) used to build table definitions

### Keys: Superkeys, Candidate Keys, Primary Keys, and Foreign Keys

*   Broad definition: a key is a subset of attributes/columns.
*   Definitions:
    *   Superkey: a set of attributes sufficient to uniquely identify a tuple (row).
    *   Candidate key: a minimal superkey (removing any attribute breaks uniqueness).
    *   Primary key: one candidate key chosen as the main identifier.
*   Notes on primary key choice:
    *   Sometimes there is a “natural” primary key.
    *   Often there isn’t, so you add a “synthetic” (surrogate) key for identification.
*   Foreign keys and constraints:
    *   A key in one table references the primary key of another table.
    *   Example: `dept_name` in `instructor` references `department`.

### Many-to-Many Relationships and Join Tables (e.g., `takes`)

*   Highlights that foreign keys naturally model one-to-many:
    *   Example: many instructors belong to one department; modeled by `instructor` holding a foreign key to `department`.
*   Many-to-many requires an intermediate table:
    *   Example: students and sections (many students per section; many sections per student).
    *   `takes` functions as the join/bridge table enabling the many-to-many relationship.
*   `takes` as an example where keys overlap:
    *   Its primary key is composite and includes:
        *   Student identifier (foreign key to `student`)
        *   Section identifier (composite foreign key to `section` via `course_id`, `sec_id`, `semester`, `year`)
*   Important modeling point: join tables are also a place to store relationship-specific attributes:
    *   `takes.grade` belongs to the student-section pairing, not solely to student or section.

### Primer on Relational Algebra: Purpose and Operators

*   Instructor introduces relational algebra as an abstract, set-theoretic view of data and queries.
*   Describes relational algebra as a procedural query language:
    *   Operators take relations as inputs and produce relations as outputs.
    *   Queries are sequences of transformations (expand/contract/alter relations).
*   Six basic operators mentioned:
    *   Selection (`σ`)
    *   Projection (`π`)
    *   Union (`∪`)
    *   Set difference (`−`)
    *   Cartesian product (`×`)
    *   Rename (`ρ`)
*   Brief discussion on Greek-letter notation:
    *   Mnemonic links (e.g., `σ` ~ “select”, `π` ~ “project”), but overall somewhat arbitrary.

### Selection (`σ`) and Its Connection to SQL `WHERE`

*   Clarifies: relational algebra “select” corresponds to SQL `WHERE`, not the SQL `SELECT` keyword.
*   Selection defined as filtering tuples/rows according to a predicate.
*   Example described:
    *   Select instructors where `dept_name = 'Physics'` (written as `σ_{dept_name = 'Physics'}(instructor)`).
*   Predicate language notes:
    *   Comparisons use standard relational operators.
    *   Predicates can be combined with conjunction/disjunction/negation (`AND`, `OR`, `NOT`).
    *   Predicates can compare two attributes (e.g., `dept_name = building`).

### Projection (`π`) and Its Connection to SQL Column Selection

*   Projection removes attributes/columns, returning only needed columns.
*   Relational algebra removes duplicates automatically after projection (set semantics).
*   Contrast with SQL:
    *   SQL does not remove duplicates unless `DISTINCT` is used.
*   Composition example described:
    *   First select physics instructors, then project only their `name` to get names of physics instructors.

### Cartesian Product (`×`) and Its Connection to SQL Comma Joins / Cross Joins

*   Cartesian product combines every tuple from one relation with every tuple from another.
*   Instructor notes it is rarely directly useful because it produces too many meaningless pairings.
*   Student observation:
    *   If you “join without a condition,” you effectively get the Cartesian product.
*   Instructor demonstrates in a live database (SQLite-like environment implied):
    *   Creates/loads the university database schema and data.
    *   Runs a query equivalent to `instructor × teaches` using SQL syntax `SELECT * FROM instructor, teaches;`
*   Notes ambiguity issues:
    *   Both tables contain an `ID` column; `SELECT ID ...` is ambiguous without qualification (e.g., `instructor.ID` vs `teaches.ID`).

### Joins as Filtered Cartesian Products and SQL Join Syntax Variants

*   Explains join conceptually:
    *   A join can be seen as `Cartesian product` followed by `selection` filtering matching keys.
*   Demonstration:
    *   Adds a `WHERE` condition such as `instructor.ID = teaches.ID` to filter the cross product to meaningful pairs.
*   SQL join syntax variants discussed:
    *   `... JOIN ... ON <predicate>`:
        *   Most flexible
        *   Does not automatically remove duplicate join columns
    *   `... JOIN ... USING (col1, col2, ...)`:
        *   Joins on same-named columns listed
        *   Collapses duplicates of the join columns in output
    *   `NATURAL JOIN`:
        *   Joins on all columns with the same name in both tables
        *   Collapses duplicates of join columns
*   Critique of `NATURAL JOIN`:
    *   Often unusable in common schema conventions (e.g., `user.id` vs `account.user_id`).
    *   Risk of accidental matches when unrelated columns share names (e.g., `name` in both `student` and `instructor`).
    *   Schema evolution hazard: adding a new same-named column can silently change join behavior.
*   Instructor notes personal tradeoff:
    *   `NATURAL JOIN` can feel “glorious” and concise when naming is designed for it, but is “treacherous” and fragile.

### A “Master Query” Mental Model: Expand → Filter → Aggregate → Select/Present

*   Instructor introduces a common strategy for constructing typical SQL queries:
    *   Expand: `FROM` + `JOIN`s to gather all needed information into a wide intermediate result.
    *   Filter: `WHERE` to restrict to the rows of interest without changing structure.
    *   Aggregate: `GROUP BY` (and aggregation functions) when multiple rows should be collapsed into one per entity/group.
    *   Then: choose output columns (`SELECT`), plus presentation steps like sorting/limiting.
*   Example problem described: “Find any student who has ever been in a class with Shankar.”
    *   Requires traversing the schema:
        *   `student` (Shankar) → `takes` → `section` → `takes` (other students in same sections) → `student` (other students)
    *   Identifies it as effectively a five-way join with table aliases/renaming needed:
        *   Two `student` instances and two `takes` instances must be distinguished.
        *   Also need a condition enforcing that the “other student” is not the same as Shankar (different `ID`).
*   Notes on building queries:
    *   Though SQL is written `SELECT ... FROM ... WHERE ...`, query *construction* often starts from `FROM`/joins, then adds filters, then decides projection/output.
    *   Relational algebra framing: each join/filter creates a new relation in stages.

### Aggregation Details: `GROUP BY` Rules and `DISTINCT` Comparison

*   Instructor explains why aggregation might be needed:
    *   A student might share multiple classes with Shankar, yielding duplicates unless collapsed.
    *   In this specific example, `DISTINCT` could be sufficient, but `GROUP BY` is shown as a general pattern.
*   `GROUP BY` rule emphasized:
    *   Any selected column must either:
        *   Appear in the `GROUP BY` list, or
        *   Be computed by an aggregation function (`COUNT`, `AVG`, `MIN`, `MAX`, etc.)
*   Discusses why grouping by `ID` alone isn’t enough if also selecting `name`:
    *   Even if `ID` functionally determines `name`, SQL requires explicit grouping to avoid ambiguous “which value to pick” issues.
*   Notes DBMS differences:
    *   SQLite is described as permissive and may allow selecting non-grouped, non-aggregated columns by arbitrarily choosing a value, but that is nonstandard and unsafe.
