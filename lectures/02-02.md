### SQL `NULL`: meaning, testing, and 3-valued logic

*   Framed `NULL` as a special value meaning “doesn’t exist” / “unknown,” not as “zero”
*   Emphasized the “funny rules” that follow from `NULL`:
    *   Arithmetic with `NULL` yields `NULL` (e.g., `5 + NULL` results in `NULL`)
    *   Comparisons involving `NULL` yield `UNKNOWN` (not `TRUE`/`FALSE`)
        *   Examples discussed: `NULL = NULL`, `NULL <> NULL`, comparisons like `5 < NULL` all become `UNKNOWN`
*   Correct way to test `NULL`:
    *   Use `IS NULL`, not `= NULL`
    *   Use `IS NOT NULL` to assert a value is present
*   Introduced 3-valued Boolean logic (`TRUE`, `FALSE`, `UNKNOWN`) and its combination rules:
    *   `TRUE AND UNKNOWN` → `UNKNOWN`
    *   `FALSE AND UNKNOWN` → `FALSE`
    *   `UNKNOWN AND UNKNOWN` → `UNKNOWN`
    *   `UNKNOWN OR TRUE` → `TRUE`
    *   `UNKNOWN OR FALSE` → `UNKNOWN`
*   Noted an important practical detail:
    *   In a `WHERE` clause, `UNKNOWN` is ultimately treated like `FALSE` (row is filtered out)
*   Guidance:
    *   Don’t necessarily memorize all rules, but recognize when `NULL` is involved, then stop, think, and verify behavior against the rules

### Aggregate functions: purpose, common functions, and `DISTINCT` inside aggregates

*   Defined aggregate functions as operating over a *multiset* of rows/values (duplicates allowed) and returning a single value
*   Listed common aggregates:
    *   `AVG`, `MIN`, `MAX`, `SUM`, `COUNT`
*   Example: average salary for instructors in a department
    *   Use `AVG(salary)` with appropriate `FROM` and `WHERE` filtering
    *   Noted the aggregate argument can be an expression (e.g., salary plus other compensation fields)
*   Example: count distinct instructors teaching in a given term (Spring 2018)
    *   Discussed why duplicates can occur when counting from a teaching/assignment table (an instructor can appear multiple times due to multiple sections)
    *   Used `COUNT(DISTINCT ID)` to avoid counting duplicates
*   Mentioned common counting idioms:
    *   `COUNT(*)` is common, but the instructor discussed historical performance concerns in some systems
    *   Demonstrated preference for counting an expression like `COUNT(1)` as a practical habit

### Console demonstration: `COUNT(*)` vs `COUNT(1)` and what `SELECT 1` means

### `GROUP BY`: producing per-group summaries and SQL’s selection rule

*   Motivated `GROUP BY` as the phase where many rows are merged into group summaries
*   Walked through an example with instructors:
    *   Start from `instructor`, select relevant columns (`dept_name`, `salary`)
    *   Consider ordering (`ORDER BY dept_name`) as an intermediate inspection step, but noted it’s not required for grouping
    *   Use `GROUP BY dept_name` to produce one output row per department
*   Highlighted a common mistake and why it matters:
    *   If you `SELECT dept_name, salary` but group only by `dept_name`, then `salary` is ambiguous
    *   Explained SQLite’s lax behavior: it may return an arbitrary salary value from each group
    *   Stated the correct standard rule:
        *   Anything in `SELECT` must be either:
            1.  Included in the `GROUP BY`, or
            2.  Produced via an aggregate function
*   Corrected version:
    *   `SELECT dept_name, AVG(salary) ... GROUP BY dept_name`
*   Brief discussion of `AVG` vs “sum divided by count”:
    *   A student noted you could compute the average manually as `SUM(...) / COUNT(...)`
    *   Instructor confirmed `AVG` exists to simplify that common pattern
*   Mentioned grouping on computed/bucketed values:
    *   You can group by expressions (e.g., truncating/dividing values to form buckets)

### JSON extensions in SQLite: generating JSON with aggregates

*   Transitioned from standard aggregates to SQLite’s JSON-related functions/extensions
*   Discussed motivation:
    *   JSON is a widely used data interchange format in web/portals
    *   Many systems store JSON blobs in columns even though it breaks strict relational normalization (“rules are made to be broken”)
    *   SQLite supports a `JSONB` (binary) representation for compactness/performance (as described)
*   Demonstrated scalar JSON construction:
    *   Example of creating a JSON array using `json_array(1,2,3)` (shown as producing correct JSON array syntax)
*   Demonstrated JSON aggregate construction:
    *   Used `json_group_array(...)` to collect values into a JSON array
    *   Used `json_group_object(key,value)` to build a JSON object mapping keys to values

### Materializing JSON results into a table and extracting them back into relations

*   Created a new table to store department-related JSON data (a “department info” style table)
    *   Discussed that SQLite would store JSON as `TEXT` (even if the schema uses a JSON-like type name)
*   Populated the table using an `INSERT ... SELECT` pattern:
    *   Used a nested `SELECT` (subquery) to generate rows for insertion rather than manually listing values
    *   Used `GROUP BY dept_name` and a JSON aggregate to store per-department JSON blobs of salaries
*   Labeled this as “materialized” data:
    *   Values were computed once and stored, rather than regenerated each query
*   Introduced table-valued JSON functions for extraction:
    *   Discussed `json_each` (and variants with paths / `jsonb` variants) as producing a relation with columns like key/value/type
*   Demonstrated how to use `json_each` by combining it in the `FROM` clause:
    *   Explained this as a cross product / join-like expansion: one row becomes multiple rows by iterating JSON members
    *   Selected columns such as the department name plus `key` and `value` to “re-extract” a relational view of the JSON object
*   Mentioned indexing implications:
    *   You can create indexes on expressions, including expressions that extract fields from JSON, to accelerate queries over JSON content
*   Suggested a preferred design pattern:
    *   Keep base data normalized
    *   Assemble JSON in SQL at query time to return to a client (instead of building JSON on the client side)

### Re-centering on aggregation: `HAVING` and query phases

*   Re-stated aggregation as another phase of query formation:
    *   Join enough tables to get needed data per row
    *   Filter rows with `WHERE`
    *   Group/aggregate with `GROUP BY`
*   Introduced `HAVING`:
    *   `HAVING` is like `WHERE` but applies *after* grouping/aggregation
    *   Example idea: select only departments where `AVG(salary) > 42000`
*   Student question: why separate keywords (`WHERE` vs `HAVING`) instead of one that changes position?
    *   Instructor response: likely clarity/readability; distinct keyword makes it easier to interpret and search for, though it may be somewhat arbitrary

### Views and `WITH` CTEs: building complex queries in stages

*   Introduced `CREATE VIEW`:
    *   A view stores a query and can be queried like a table (`SELECT`/join against it)
    *   SQLite computes views dynamically (contrasted with materialized views in other DBs)
*   Presented a complex view: “time slots available to sections”
    *   Goal: return a relation like (department, section, time slot, time slot priority) after considering:
        *   Faculty availability and penalties
        *   Section time-slot constraints
        *   Intersection of constraints (faculty must be available; section must allow the slot)
        *   Combine penalties by taking the most severe (worst) applicable penalty
    *   Mentioned complications:
        *   Sections may have multiple instructors (or zero), requiring careful combination logic
*   Introduced `WITH` as common table expressions (CTEs):
    *   Used to construct a complex query in named stages (temporary subqueries)
    *   Example structure described:
        *   A first stage like `per_faculty`
        *   Subsequent stages like grouped/per-faculty aggregates and counts
        *   A final query consuming these intermediate named relations
*   Discussed specific techniques inside the view:
    *   `GROUP BY` combined with aggregates like `MIN(...)` to collapse multiple ways a constraint could apply
    *   Use of conditional logic (`CASE WHEN ...`) to handle `NULL` priorities safely
        *   Specifically noted: “can’t just take `MIN` naively when `NULL` is involved”; used `CASE` + `IS NULL` checks
    *   Motivation for grouping:
        *   The same (section, time slot) might be derived multiple ways via tags, so grouping consolidates duplicates and selects the most severe penalty

### Additional scheduling queries: room availability, joins, and when not to join “all the way”

*   Discussed a room-availability style query:
    *   Goal: compute per (department, section, room) a priority/penalty
    *   Joined from sections-to-be-scheduled through section-room tags and room tags
    *   Used `GROUP BY dept, section, room` to merge duplicates caused by overlapping/non-disjoint tags
    *   Used an aggregate (e.g., `MIN`) to choose the worst priority among multiple tag-derived paths
*   Explained a join-graph optimization mindset:
    *   Sometimes you do not need to join all the way to an endpoint table if the needed attribute is already present in the adjacent/join table
    *   Gave the earlier “students who had a class with Shankar” query as an analogy:
        *   You can often “pull in the ends” and avoid unnecessary joins if the needed fields are already available
*   Contrasted with a query where joining to the `room` table is necessary:
    *   Example: “rooms used by departments” needed room capacity, so it required joining to the room table

### Complexity example: conflicts, multiple joins, and materializing expensive results

### Nested subqueries: where they appear and performance cautions

*   Transitioned from CTEs (“serial queries”) to nested subqueries
*   General guidance:
    *   Avoid nested subqueries when a join would work (especially while learning)
    *   Nested queries can encourage imperative thinking (like writing nested loops) rather than declarative SQL
*   Noted where subqueries can appear:
    *   In `FROM` (subquery-as-table): `FROM` doesn’t have to reference disk tables; it can reference query results
    *   In `WHERE` (subquery-as-filter): be careful about subqueries that execute per row
    *   In `SELECT` (subquery-as-expression) and as part of aggregate-related calculations
*   Reconnected to relational algebra viewpoint:
    *   Every query produces a table (relation) which can feed another query
    *   SQL transformations are relation-to-relation until a final result is returned or stored

### Subqueries with `IN` and revisiting set concepts

*   Mentioned set operators again and noted SQLite limitations (many such operators/features are not fully supported there)
*   Demonstrated the `IN` pattern as a set-membership test:
    *   Example: courses offered in both Fall 2017 and Spring 2018
        *   Outer query selects distinct course IDs from fall sections
        *   Uses `course_id IN (subquery selecting spring course_ids)`
    *   Noted `IN` is also commonly used with a literal list of values (e.g., `name NOT IN (...)`) as a convenience over chaining comparisons
*   Another nested-subquery example:
    *   Counting distinct students who took sections taught by an instructor with a given ID
    *   Inner query finds the instructor’s taught sections (course/section/term identifiers)
    *   Outer query finds matching rows in `takes` and counts `DISTINCT` student IDs
*   Optimization discussion:
    *   With joins, query optimizers can reorder and choose efficient execution plans using statistics and indexes
    *   CTEs and some nested-query patterns can act as “barriers” that reduce optimizer freedom (depending on DB)
    *   Reinforced preference for declarative formulations (“what data I want”) rather than telling the DB “how to do it”

### `SOME`/`ALL`, `EXISTS`, and correlated subqueries (and why to watch them)

*   Discussed `SOME`/`ANY`-style comparisons (not supported in SQLite, per the lecture):
    *   Example idea: find instructors with salary greater than `SOME` salary of instructors in Biology
    *   Equivalent join-based formulation described conceptually as a cross product of instructors with biology instructors plus a comparison and `DISTINCT`
*   Discussed `ALL`:
    *   Example idea: salary greater than `ALL` biology instructor salaries
    *   Mentioned alternative: compare against `MAX(salary)` in biology via an aggregate (another possible rewrite)
*   Introduced `EXISTS` / `NOT EXISTS`:
    *   Used for testing whether a relation/subquery result is empty
*   Correlated subquery example (courses offered in both terms):
    *   Outer query scans sections in Fall 2017
    *   Inner `EXISTS` subquery scans Spring 2018 sections *but references the outer query’s `course_id`*
    *   Explained why correlation matters:
        *   The inner query depends on the current outer row, so it may effectively run per outer row
        *   This is the pattern that can cause performance problems on large datasets
*   Correlated `NOT EXISTS` example (relational division pattern):
    *   Task: find students who have taken *all* courses offered in the Biology department
    *   Strategy described:
        *   Select students
        *   For each student, check there does *not* exist a Biology course that is “missing” from that student’s taken set (often expressed using an `EXCEPT`-style difference)
    *   Class discussion on alternative join-based approaches:
        *   Suggested using joins and aggregation/counting to compare “number taken” vs “number offered”
        *   Mentioned outer joins as a technique to find “missing matches” (rows that would become `NULL` on the non-matching side), though the instructor noted the first attempt would need careful setup
*   Reinforced the point:
    *   Correlated subqueries are not “never,” but they are the main ones to watch for performance issues and optimizer limitations
