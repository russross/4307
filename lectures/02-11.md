## Indexing and B-Tree/B+Tree Storage in SQLite

*   Transition from discussing normalization forms to the “big picture” of how databases execute queries efficiently via indexing
*   Core B-tree concepts: block/page-sized nodes, high fan-out, balanced depth, and how inserts cause splits that grow the tree upward
*   Search behavior for point lookups and range scans, emphasizing disk I/O cost vs. in-memory work and why upper tree levels tend to stay cached
*   B+ trees vs. B-trees in SQLite: tables (by default) use B+ trees keyed by an internal integer `rowid`, while secondary indexes are B-trees
*   Practical index construction in SQLite using `CREATE INDEX` and interpreting `EXPLAIN QUERY PLAN`, plus tradeoffs/overheads and the need to measure before optimizing

### What an index is and why it matters
*   Introduces indexing/index construction in more detail (building on a brief earlier introduction)
*   Key idea: base table storage has *some* physical order (sometimes insertion/random; later, SQLite specifics)
*   Defines an index as:
    *   A copy of some data (or parts of rows) sorted by a different key
    *   Used to efficiently look up an attribute and find matching rows
*   Notes that indexing efficiency relies heavily on tree structures (B-trees/B+ trees)

### B-tree structure (balanced, wide, shallow)
*   Clarifies a B-tree is balanced but not binary
*   Diagram-level description:
    *   A root node with multiple keys and child pointers
    *   Keys partition the key space (e.g., `< 7`, `7..16`, `> 16`)
*   Node capacity is driven by fixed container size (a disk “block/page” size), not a fixed number of keys
    *   Example assumes 4 elements per node for simplicity
*   Balanced property: all leaf nodes are at the same depth
*   Performance intuition:
    *   Deeper trees are more expensive (more node visits / more I/O)
    *   B-trees are designed to be very wide (high fan-out) and thus shallow
    *   Larger block/page size generally yields flatter trees

### Pages/blocks and the “B” in B-tree (including the naming anecdote)
*   Discusses SQLite page size history:
    *   Older default mentioned as `512` bytes
    *   More modern default often `4K`, aligning with typical OS/disk transfer granularity
*   Mentions “B” is often said to mean “block,” but there are conflicting stories
*   Shares the well-known anecdote about naming (Rudolf Bayer / Boeing / “balance” / author initial), concluding the origin is intentionally ambiguous

### Internal vs. leaf node layout
*   Notes internal nodes store keys plus child pointers
*   Leaf nodes have no child pointers, so their structure differs and can sometimes fit more entries
*   Mentions child pointers are usually small (often just integers), so they don’t dominate space

### Insertions: splitting nodes and growing the tree upward
*   Uses a visualizer/animation to illustrate insertion
*   When inserting into a full node:
    *   Split the node (roughly half the keys to left, half to right)
    *   Promote a separator key to the parent
*   If there is no parent (splitting the root):
    *   Create a new root
    *   This is how the tree increases depth
*   Emphasizes an important invariant:
    *   Depth is added at the top (root splits), not by extending the bottom
*   Mentions “bubbling up” effect:
    *   Splitting a leaf can force an insert into the parent
    *   Parent may split, cascading upward to the root

### Keeping nodes “dense enough” while allowing cheap inserts
*   Splits create slack space, giving room for future inserts
*   Tradeoff described:
    *   Too much empty space wastes storage and can increase tree height
    *   Some free space is useful so inserts don’t constantly trigger splits

### Deletions and why databases often avoid aggressive rebalancing
*   Instructor considers delete operations in the visualizer:
    *   Simple delete can leave a gap / reduce occupancy
    *   Underfull nodes could theoretically be fixed by rotation/merge
*   Notes that in practice many databases:
    *   Do not frequently rebalance/merge on deletes during normal operation
    *   May tolerate underfull or even empty pages rather than doing expensive reshuffling
*   Compaction is often done via a separate operation such as `VACUUM`
    *   Typically rebuilds the table by copying data to a new packed structure rather than “in-place” fixes

### Searching for a single key (point lookup)
*   Walks through a lookup example (e.g., searching for `315`)
*   Process:
    *   Start at root, choose child pointer based on key comparisons
    *   Repeat down to leaf
    *   Within a node, either linear scan or binary search over keys
*   Cost model emphasized:
    *   Loading a block/page from disk is much more expensive than comparing keys in memory
    *   Once a page is in memory, scanning/searching within it is relatively cheap

### Range queries and scanning through the tree
*   Example: retrieving all keys from `300` to `500`
*   Strategy described:
    *   First find the starting key (e.g., `300`) like a normal search
    *   Maintain the “trail” / “spine” of visited internal nodes (like a cursor or recursive call stack)
    *   Iterate forward through leaf-level keys, and when a leaf is exhausted, move back up and continue to the next subtree
*   Emphasizes practical efficiency:
    *   Even though traversal moves up and down, the relevant internal pages are already in memory
    *   The scan avoids unnecessary page reloads by keeping upper levels cached
*   Notes that a B-tree supports:
    *   Full scan from beginning to end
    *   Scanning backward from the end
    *   Scanning from an arbitrary starting key
    *   Range scans efficiently

### Caching behavior and typical performance assumptions
*   Discusses that many modern systems can have memory as large as (or larger than) the database
    *   After warm-up, many pages may be resident in cache
*   More typical assumption: disk larger than memory; database is large
*   Observed caching pattern:
    *   Root and intermediate levels tend to remain cached even for large tables
    *   Leaf pages are what churn more often
*   Consequence:
    *   With warm cache, a typical lookup may effectively cost ~one page read (leaf), since upper levels are memory-resident

### Disk/page granularity and why page-sized nodes matter
*   Confirms each node corresponds to a fixed-size page/block (often `4K`)
*   Notes typical I/O behavior:
    *   Even if you want a single byte, disk/OS interfaces transfer a full block/page
*   Reinforces “tuned to the storage layer” theme: B-trees are designed around block-based I/O

### B+ trees: what changes and why SQLite uses them for tables
*   Introduces B+ trees as “similar but with visible differences”
*   Main distinction:
    *   Full row data is stored only in leaf nodes
    *   Internal nodes store only keys (often duplicates of leaf keys) to guide search
*   Explains that internal keys partition ranges (e.g., `< 3`, `3.. < 5`, `>= 5`) and boundary conventions can vary (`<` vs. `<=`)
*   Mentions a common B+ tree depiction includes linked leaf nodes for efficient sequential scans
    *   Notes SQLite does not use explicit leaf-to-leaf links; it instead relies on cursor/spine traversal as described earlier

### SQLite `rowid` tables and why integer keys make B+ trees attractive
*   SQLite table storage details:
    *   If a table declares an `INTEGER PRIMARY KEY`, SQLite uses it as the row identifier (effectively the `rowid`)
    *   If not, SQLite still assigns a unique internal row identifier
    *   Conceptually there is always a hidden `rowid` column unless using `WITHOUT ROWID`
*   Why B+ trees fit tables in SQLite:
    *   Keys replicated in internal nodes are always integers (`rowid`), which are small
    *   Internal pages become extremely dense (high fan-out)
    *   Leaf pages hold full records (potentially large), so they are less dense
*   Benefit: very shallow trees due to dense upper levels, improving lookup efficiency

### Why SQLite uses B-trees for secondary indexes (key/value size tradeoff)
*   Returns to the contrast: indexes in SQLite are B-trees
*   Discusses a typical index scenario (e.g., movies indexed by title):
    *   Index key: the indexed attribute (e.g., `title`), potentially large (strings)
    *   Index value/payload: the corresponding `rowid` / primary key, small (integer)
*   Explains why B+ tree replication is unattractive for indexes:
    *   B+ trees replicate keys in internal nodes while keeping “full data” only in leaves
    *   For indexes, the *key* is the large part; replicating it at multiple levels is expensive
    *   The part you would avoid replicating (the row pointer/`rowid`) is small anyway
*   With a B-tree index, the structure stores the necessary key/pointer records throughout the tree without the B+ style “replicate big keys in internal nodes” overhead

### `WITHOUT ROWID` tables: opting out of the synthetic `rowid`
*   Notes SQLite supports `WITHOUT ROWID` on table creation:
    *   Placed after the `CREATE TABLE` definition (before the terminating semicolon)
    *   Causes the table to be stored keyed by the declared primary key rather than a synthetic `rowid`
    *   Implemented more like an “ordinary B-tree” keyed by that primary key
*   Motivation: storing rows physically ordered by a natural primary key can improve range queries on that key
*   Example discussed (course schedule dataset):
    *   Primary key like `CS 4307` with other columns (title, etc.)
    *   If stored keyed by course identifier, range/prefix scans (e.g., all courses with `CS` prefix) become naturally efficient
*   Contrasts with default `rowid` ordering:
    *   Row IDs often reflect insertion order; data can be effectively scattered with respect to natural keys
    *   Without a secondary index, queries like “all keys with prefix `CS`” devolve to a full table scan

### The “failure mode” without the right index: linear scans
*   Reinforces that without an index, the engine’s options are limited:
    *   Use the table’s physical key order (whatever the storage structure is keyed on)
    *   Or scan the entire table
*   References earlier discussion about deletion constraints/foreign-key-like checks:
    *   If you need to find “who references this record” without an index, you may have to scan the whole table each time

### Secondary indexes as a two-step lookup
*   Describes how a secondary index helps when the table is keyed by `rowid`:
    *   Scan the secondary index in key order (e.g., course identifiers)
    *   Each index entry yields a `rowid`
    *   Then perform a lookup in the primary table by that `rowid`
*   Notes this is still efficient, but it’s fundamentally a two-step process
    *   Mentions possible optimizations (e.g., batching/sorting row IDs), but the indirection remains

### SQLite demo: `EXPLAIN` / `EXPLAIN QUERY PLAN` and creating an index
*   Instructor switches to scheduling data and shows a `courses` table (noting it was created with `WITHOUT ROWID`)
*   Runs a query selecting `course` and `course_name`, then a query with a `WHERE course_name = 'Database Systems'`
    *   Notes performance is fast mostly because the dataset is small
*   Uses `EXPLAIN` to show SQLite compiles a query into a low-level virtual machine program (bytecode-like, with numeric parameters)
*   Uses `EXPLAIN QUERY PLAN` to get a higher-level description
    *   Initially shows a plan that scans `courses` (full table scan)
*   Creates an index:
    *   Uses `CREATE INDEX ... ON courses(course_name)` (including naming the index and specifying the indexed column list)
*   Reruns `EXPLAIN QUERY PLAN` and observes the plan changes:
    *   Now it searches using the index rather than scanning the full table
    *   Notes “covering index” behavior: if the query only needs columns present in the index (including the table’s primary key), SQLite can answer from the index without reading the base table
*   Explains lookup mechanics:
    *   Index maps `course_name` to the table’s primary key (`course` in this schema)
    *   If the query needs other columns not in the index, the engine uses the primary key to fetch the full row from the table

### Indexes as managed redundancy (indexing as a kind of denormalization)
*   Frames indexing as introducing redundancy:
    *   Normalization: store facts once and link them
    *   Indexes: store extra copies of selected fields in alternate orders for fast access
*   Distinguishes this from “manual denormalization” in key-value stores:
    *   Key-value designs may embed anticipated query structure in keys/values (e.g., precomputing timelines)
    *   Indexes similarly anticipate access paths, but remain managed by the DBMS
*   Emphasizes indexes are an optimization layer:
    *   Can be created/dropped without changing table data model
    *   Query planner may even create temporary indexes for specific queries when it’s worthwhile

### Costs of indexes and how to decide what to index
*   Every index has overhead:
    *   Space overhead from duplication
    *   Write overhead on `INSERT`, `UPDATE`, and `DELETE` (must update base table plus all affected indexes)
*   Advises against “spraying indexes everywhere”
*   Where to start when a database is slow:
    *   Heuristics: likely lookup fields (names, commonly searched attributes)
    *   Index join keys: primary keys and foreign keys used in joins
    *   Watch out for systems where the declared primary key is not the physical key (e.g., `rowid` behavior), causing unexpected scans unless you add an index or change table storage strategy

### Measuring and diagnosing: don’t guess, profile
*   General optimization principle: measure first
*   Find representative workloads/realistic traces that expose slowness
*   Identify slow queries:
    *   Many DBMSs (e.g., MySQL, PostgreSQL) can log queries slower than a threshold (e.g., 500 ms)
    *   Alternatively, measure at the application layer
*   Use tools like `EXPLAIN QUERY PLAN` to see what the engine is doing and why
*   Requires developing intuition about:
    *   What’s fast vs. slow
    *   What access paths are possible with existing indexes and storage layouts

### SQL as a “leaky abstraction” and why implementation knowledge matters
*   Discusses Joel Spolsky’s “Law of Leaky Abstractions” and SQL as a classic example
*   Ideal SQL experience:
    *   Write declarative queries describing *what* data you want
    *   DBMS decides *how* to execute efficiently
*   Reality:
    *   Sometimes the optimizer doesn’t choose what you expect
    *   To improve performance (e.g., by adding the right index), you must understand current execution behavior and available mechanisms
*   Sets expectation that much of the rest of the semester will focus on how databases work internally to build that expertise

### Other indexing structures and why B-trees remain dominant in practice
*   Mentions other index structures (e.g., hash indexes):
    *   Great for exact-match lookups
    *   Poor for range/prefix queries (e.g., “starts with `CS`”)
*   Notes SQLite largely relies on B-tree/B+ tree structures and still performs very well for many workloads
*   Comments on hash-table theoretical advantage (`O(1)` lookup) vs. practical considerations:
    *   With cache-resident upper tree levels, B-tree access often behaves like “one main page read” in practice
    *   Modern systems often have large RAM relative to data size, reducing the advantage of pure hashing for many cases

### Memory sizing, OLTP, and industry perspective (Stonebraker discussion)
*   Mentions Michael Stonebraker’s perspective on database market segments:
    *   Data warehousing vs. OLTP (online transaction processing)
    *   Column stores for analytics/warehousing workloads
*   For large OLTP databases (e.g., terabyte scale), argues RAM may be affordable enough to keep much or all of the working set in memory
*   Notes a viewpoint that if you assume data is in RAM, you can redesign DBMS from scratch for major speedups (some such systems exist, adoption slower than expected)

### B-trees as an in-memory data structure (cache hierarchy motivation)
*   Observes B-trees aren’t only for disk:
    *   Modern CPUs have L1/L2/L3 caches; main memory is “slow” relative to caches
    *   Hardware moves data in cache lines (e.g., ~64 bytes), not single bytes
*   Motivation for in-memory B-trees:
    *   Higher fan-out reduces pointer-chasing depth compared to balanced binary trees (e.g., red-black trees)
    *   Better locality: more comparisons within a contiguous node can be faster than many scattered pointer dereferences
*   Mentions Rust’s standard library includes B-tree-based collections
    *   Notes an implementation detail: node sizes chosen empirically (on the order of ~12–13 entries per node), based on performance testing
