### Normalization Motivation and Historical Context (Codd)

*   Introduced “database normalization” (relational model design rules) and attributed early normal forms to Edgar F. Codd (circa 1970)
*   Framed normalization’s core goals:
    *   Minimize/eliminate certain redundancies
    *   Avoid hazards/anomalies that arise from poor structure
    *   Provide a standard approach to structuring data in an SQL database
*   Quoted/outlined Codd’s goals for first normal form (as overall design goals):
    *   Free relations from undesirable insertion, update, and deletion dependencies (anomalies)
    *   Reduce the need to restructure relations as new data types are introduced, increasing application lifespan
    *   Make the relational model more informative to users
    *   Keep relations “neutral” to query statistics/patterns that change over time
*   Clarified “undesirable insertion/update/delete dependencies” using the earlier in-course example (albums/artists in a flat-file design):
    *   Updating band information required updating many rows (e.g., every song), creating inconsistency risk
    *   Normalization aims for “surgical” updates rather than cascading manual fixes
*   Interpreted the “lifespan” point as decoupling the data model from current usage/query patterns:
    *   Model data in an elemental/abstract way based on intrinsic relationships
    *   Avoid structuring tables purely around expected queries

### Query-Pattern Neutrality: Redis/Datastore vs PostgreSQL, and Twitter

*   Shared a personal side-project experience as an illustration of data evolution and flexibility:
    *   Initially implemented storage in Redis (`Redis`) as a RAM-based key-value store with richer value types (strings, hashes, lists, queues)
    *   Used server-side `Lua` scripts to serialize multi-step operations, approximating atomic/transaction-like behavior by running scripts sequentially
    *   Found it painful for ad hoc questions: needed new scripts for each query
*   Moved to Google App Engine/Google Cloud Datastore (non-relational/limited relational, designed for scalability):
    *   Required planning around implementation efficiency/scalability
    *   Ad hoc querying was worse: required adding code, compiling, redeploying, then removing endpoints
*   Finally switched to PostgreSQL (`PostgreSQL`/Postgres):
    *   Valued “boring” relational modeling for flexibility and easy ad hoc SQL queries
*   Connected to the “optimize for common case” tradeoff via a Twitter example:
    *   Timeline fan-out: on tweet, write into followers’ timelines to make reads fast (dominant traffic)
    *   Posting becomes more expensive; writes are less frequent than reads
    *   Real-time search/streaming requires different layout/pipeline; Twitter built parallel systems for that
*   Concluded that specialized/denormalized layouts can be appropriate at scale, but relational modeling is usually best for flexibility; performance is a separate (often solvable) concern

### Anomalies in Unnormalized Data (Insertion/Update/Deletion)

*   Distinguished terminology:
    *   “Unnormalized” data: not yet normalized
    *   “Denormalized” data: deliberately relaxing normalization for some reason
*   Gave classic anomaly examples:
    *   Insertion anomaly:
        *   Table with `faculty_id`, name, hire date, and `course_code`
        *   Hiring new faculty with no assigned course becomes impossible if `course_code` is required
    *   Update anomaly:
        *   Employee skills table with `employee_id`, address, and skill
        *   Updating address in one row but not another yields contradictory addresses for the same employee
        *   The issue is not “wrong” data per se, but allowing contradictions due to structure
    *   Deletion anomaly:
        *   Deleting the last course row for a faculty member deletes the faculty information entirely
*   Summarized normalization as the method to remove these hazards and reduce redesign when extending a database

### Normal Forms Roadmap and Hierarchical Nature

*   Listed the progression and dates:
    *   1NF (Codd, 1970)
    *   2NF and 3NF (Codd, 1971)
    *   Boyce–Codd Normal Form (BCNF) (Codd + Raymond Boyce, 1974)
    *   Mentioned higher normal forms exist but are more specialized; “doing pretty well” by BCNF
*   Addressed student confusion about “first/second/third”:
    *   Confirmed they are hierarchical: 2NF implies 1NF, 3NF implies 2NF, etc.
    *   Described it as progressively tightening constraints

### Worked Example Setup: Primary Keys and an Unnormalized “Book” Table

*   Introduced a single, unnormalized book/publishing record containing many attributes in one table:
    *   `title`, author, author nationality, `format`, `price`, multiple `subject` values, pages, publisher, publisher country, `genre_id`, genre name, etc.
*   Reviewed primary key basics:
    *   A relation/table needs a primary key that uniquely identifies a row
    *   Choosing keys relies on domain knowledge and identifying candidate keys
    *   Noted goal of selecting a minimal candidate key
*   In the example, identified a composite primary key:
    *   (`title`, `format`) distinguishes hardcover vs paperback editions

### First Normal Form (1NF): Atomic/Scalar Fields

*   Stated 1NF rule: every field contains a single scalar (atomic) value (no sets, no nested records)
*   Identified the violation: `subject` contained a list of subjects (multiple values in one field)
*   Clarified “scalar/atomic” terminology:
    *   Student associated “scalar” with “a number” (linear algebra usage)
    *   Instructor clarified: here it means “one thing” (not a collection like a vector)
*   Showed the fix conceptually:
    *   Split subjects into a separate relation (e.g., a `title_subject` mapping table)
    *   Join the book table with the subject table to reconstruct the original combined view
*   Discussed key/foreign key nuance in this decomposition:
    *   Original primary key was (`title`, `format`)
    *   Subject mapping referenced only `title` (not `format`) because subjects don’t change across formats
    *   Noted this is less common than mirroring the full primary key, but avoids duplicating subjects per format edition
*   Warned against assuming normalization (extra joins) will necessarily be too slow:
    *   Called out “premature optimization” concerns
    *   Emphasized correctness/modeling first; optimize later if needed

### Second Normal Form (2NF): Eliminating Partial Dependencies

*   Stated the 2NF idea informally:
    *   When using a composite key, every non-key attribute must depend on the whole key, not just part of it
*   Used the book example to illustrate:
    *   Composite key is (`title`, `format`)
    *   Many attributes (author, nationality, publisher, etc.) depend only on `title`
    *   `price` depends on both `title` and `format` (hardcover vs paperback pricing)
*   Explained why this is a problem:
    *   Redundant repeated data across multiple rows (most columns duplicated across formats)
    *   Update anomaly risk (e.g., update author nationality in hardcover row but not paperback row)
    *   Mentioned potential storage overhead, but emphasized correctness/consistency as the primary motivation
*   Described the conceptual fix:
    *   Create a `book` table keyed by `title` for title-dependent attributes
    *   Create a separate `price`/edition table keyed by (`title`, `format`) for format-dependent attributes like `price`
*   Related to earlier course example (university `section` composite key):
    *   `section` key included multiple attributes (e.g., course, section, year, semester)
    *   Non-key attributes like `time_slot_id` depend on the full combination, not a subset

### Third Normal Form (3NF): Eliminating Transitive Dependencies

*   Noted that even after 2NF, redundancies remained:
    *   `genre_id` and genre name together
    *   `author` and author nationality
    *   publisher and publisher country
    *   Potential multi-author-per-book considerations (raised by a student)
*   Identified the dependency issue as transitive dependency:
    *   Genre name depends on `genre_id`, and `genre_id` depends on `title` (so genre name depends transitively on `title`)
    *   Author nationality depends on author; author depends on title
    *   Publisher country depends on publisher; publisher depends on title
*   Described the 3NF decomposition outcome:
    *   Keep `book` with attributes that depend directly on the key (e.g., `title`, `author`, pages, “thickness”, publisher, `genre_id`)
    *   Keep `price` table for (`title`, `format`) → `price`
    *   Create separate lookup/entity tables:
        *   `author` → nationality
        *   publisher → country
        *   `genre_id` → genre name
*   Commented that by 3NF the structure “looks better organized” and avoids many inconsistencies
*   Had a brief aside about what “thickness” might mean in publishing (uncertain; possibly a classification tied to paper/packaging/postal constraints)

### Formal Definitions and Terminology Clarifications (Wikipedia, ACID, NoSQL)

*   Responded to student difficulty with formal definitions by revisiting them after the examples
*   Read/outlined a formal 1NF definition:
    *   Relation is in 1NF if each attribute value is atomic (single value), not a set or nested relation
*   Clarified “atomic” vs ACID atomicity:
    *   1NF atomicity: indivisible attribute values in a schema
    *   ACID atomicity: a sequence of operations treated as a single all-or-nothing transaction
    *   Explained word choice: “atomic” in general means indivisible
*   Read/outlined a formal 2NF definition:
    *   In 1NF and contains no partial dependencies
    *   A non-prime attribute must not depend on a proper subset of any candidate key
*   Read/outlined a formal 3NF definition:
    *   In 2NF and has no non-key (transitive) dependencies
    *   Non-prime attributes must not depend on other non-prime attributes
*   Mentioned practical violations:
    *   Standard SQL historically didn’t support table-valued columns, but many modern systems permit 1NF violations
    *   `JSON` fields inside text columns can encode nested structures, effectively bypassing 1NF
*   Discussed NoSQL/document stores and 1NF:
    *   Mentioned MongoDB (`MongoDB`) as document-oriented (key → JSON-like document), which encourages nested values
    *   Characterized it as often chosen for scalability and distributed efficiency, with skepticism about hype vs reality (especially historically)
*   Briefly revisited the earlier theme: “ask an LLM vs read a webpage,” noting that neither is automatically trustworthy

### Real-World Example: University Schedule CSV as a “Pre-Joined” Report

*   Observed that CSV/spreadsheet data distribution is common (“the world is a spreadsheet”):
    *   Simple and widely supported
    *   Often a poor model for real relational data
    *   Frequently represents a joined report exported from a relational system (convenient but redundant)
*   Opened an actual scheduling spreadsheet/CSV (fall schedule) and scanned columns
*   Discussed key identification:
    *   The sheet represents many sections, not faculty records
    *   A plausible primary key is `CRN` (a synthetic identifier), though noted real data exceptions (duplicate `CRN`s in biology due to special arrangements)
*   Pointed out repeated faculty attributes per section:
    *   `faculty_id`, last name, first name, middle initial, position repeated across every section taught
    *   Interpreted as a normalization issue (transitive dependencies): faculty name depends on `faculty_id`, not on the section/`CRN`
    *   Noted that in the source database these likely live in separate tables and are joined for reporting
*   Examined course identifiers and titles:
    *   `subject`, course number, section number, and course title
    *   Course title is not truly dependent on `CRN`; it depends on the course identity (subject + course number)
    *   Suggested need for a separate `course` table with `section` referencing it
    *   Noted messy cases like “special topics” where the title may vary by section (exceptions complicate strict dependency assumptions)
*   Discussed handling exceptions in real systems:
    *   Suggested patterns such as:
        *   An override field that is `NULL` unless a special title applies
        *   A separate exceptions table that is joined in when needed
        *   A `VIEW` to hide conditional override logic and present a clean interface
*   Mentioned scheduling-specific logic and exceptions from the instructor’s scheduling system:
    *   Counting number of sections to decide how strongly to penalize time conflicts (e.g., concurrent-eligible courses)
    *   Example exception: multiple sections of senior project (`CS 4600`) exist but are taught at the same time, so a naive “multiple sections reduces conflict penalty” rule can be wrong
    *   Implemented a small exceptions table (few rows) and combined discovered aggregates with explicit overrides via conditional logic in queries (e.g., `GROUP BY` + `CASE`-style logic)
*   Looked at additional schedule fields (days, start/end times, capacity, enrollment, waitlist, building, room):
    *   Considered whether `building` and `room` indicate 3NF issues; concluded you typically need both as identifiers
    *   Noted room capacity would depend on (`building`, `room`) rather than on the section; however, the sheet includes a capacity field that may be manually edited in practice
*   Concluded that real data is messy and CSV extracts are often intentionally denormalized/pre-joined for distribution, trading redundancy for convenience
