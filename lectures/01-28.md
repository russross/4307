## From Streaming Algorithms to Relational Modeling: Designing a Library Database Schema

*   Recapped the guest lecture on stream processing techniques (e.g., Bloom filters and distinct counting) and discussed how these relate to database systems.
*   Clarified the intuition behind the Flajolet–Martin distinct-count estimation algorithm, including trailing-zero maxima, probabilistic rarity, and variance reduction via multiple hashes and median-of-means.
*   Motivated approximate data structures with real-world “good enough” estimation examples (e.g., real-time traffic inference for mapping applications).
*   Built a relational model for a library system, iteratively normalizing entities and fixing many-to-many relationships with join tables.
*   Implemented the schema in SQLite, generated test data with an LLM, and wrote multi-join SQL queries (including grouping and counting) while noting practical pitfalls (keys, dates, and historical lending records).

### Clarifying the Flajolet–Martin distinct counting algorithm (guest lecture Q&A)

*   Student asked about the “Flajolet–Martin” (transcribed as “flow jar in Martin”) algorithm and what it measures.
*   Reframed the goal: estimate the number of unique elements seen in a stream without storing all elements (or even their hashes) for an exact count.
*   Core intuition:
    *   Hash values behave like reproducible random bitstrings per input.
    *   Many trailing zeros is a rare event; the rarer the event, the more items you typically must observe before seeing it.
    *   The maximum number of trailing zeros observed relates probabilistically to the number of distinct elements.
*   Explained why the estimate is roughly `2^R` where `R` is the maximum trailing-zero count encountered.
*   Discussed refinements to reduce error/variance:
    *   Use multiple hash functions / parallel sketches to avoid only producing powers-of-two and to reduce sensitivity to outliers.
    *   Combine results via averaging; then use robust aggregation (described as “median of means” / “median of averages”) to squash outliers.
*   Emphasized trade-offs: efficiency vs precision; for huge streams (billions/trillions), approximate answers are often sufficient and enable constant memory usage.

### Real-world motivation for approximate/streaming approaches: mapping and traffic

*   Student connected approximate estimation to how map apps detect congestion (yellow/red slow zones).
*   Clarified data sources for real-time traffic:
    *   Not primarily “the government” for real-time conditions.
    *   Inferred from phones and, significantly, from cell providers (handoffs between towers can help infer movement/velocity).
    *   Providers may combine multiple sources (user reports, partnerships with states/municipalities, satellite imagery processing) for road network updates.
*   Pointed out that highly precise data may not improve the user experience beyond a good approximation (e.g., “about 15 extra minutes”).
*   Discussed economics of map data:
    *   Mentioned an estimate (possibly dated) of about `$1B/year` to keep mapping data globally updated.
    *   Explained why built-in car navigation historically cost a lot: fewer users to amortize mapping costs vs phone platforms spreading cost across billions.
*   Anecdote about outdated in-car navigation failing to reflect a redesigned highway exit; noted even major providers can lag on traffic pattern changes.

### Bloom filters and “rediscovery” cycles in CS research

*   Student noted Bloom filters originated around the 1970s, were “rediscovered” in the 1990s systems research wave, and are now common in data science contexts.
*   Commented that computer science often reinvents ideas from the 1960s/1970s.
*   Mentioned a practical reason: older work may not be digitized/searchable; if it isn’t easily found via web search, it’s often overlooked.

### Transition to a database modeling exercise: library system requirements

*   Announced an in-class example to work through modeling and schema development for a “library” (physical books library, not a software library).
*   Brainstormed book attributes for an initial `books` table:
    *   Genre, author, title, publication date(s) (original vs edition), number of pages, cover image, ISBN (transcribed as “ISPN/ISVN”).
*   Noted an initial naive design: put book attributes in one table with ISBN as primary key.
*   Began improving/normalizing:
    *   Recognized authors aren’t just a single attribute of a book; need an `authors` table (name, birth date, optional link such as Wikipedia).
    *   Discussed author identifiers:
        *   Names aren’t unique; wished for a natural unique ID but settled on a surrogate `author_id`.
        *   Joked about using SSNs/barcodes; acknowledged privacy/appropriateness issues.
*   Identified relationship cardinality:
    *   Books can have multiple authors; authors can write multiple books → many-to-many.
    *   Need an intervening “join table” (called “adjoining/intervening” table; ultimately modeled as `book_authors` / similar).
    *   Removed `author_id` from `books` since there is no single correct value per book.
    *   Mentioned relationship attributes could live in the join table (e.g., “first author” ordering for academic works).

### Modeling inventory and lending (copies, patrons, and loans)

*   Added library inventory considerations:
    *   Need a notion of physical (and potentially digital) copies.
    *   Digital lending still behaves like limited “copies” due to licensing (example: Libby e-book waiting lists).
*   Debated whether to store a simple quantity vs one row per copy:
    *   Chose “one entry per copy” to track who a specific copy is lent to.
*   Introduced additional entities and attributes:
    *   `copies`: linked to `isbn`, plus a `copy_number`/local identifier; mentioned possible “condition” field as an extension.
    *   `patrons`: card number/ID, name, address (for local/tax eligibility), possibly balances/fines.
    *   `lending` (also discussed as “loans”): join between `patrons` and `copies`, with attributes like `due_date`.
*   Discussed potential further normalization (e.g., a generic `people` table shared by authors and patrons) but deferred to keep scope manageable.
*   Emphasized that adding join tables doesn’t “restrict access”; it just means queries require joins.

### Writing the SQL schema (SQLite-focused) and design decisions

*   Began drafting `CREATE TABLE` statements (starting with `books`).
*   Naming convention discussion:
    *   Preferred plural table names (`books`) because `SELECT ... FROM books` reads naturally.
    *   Mentioned Ruby on Rails / ActiveRecord conventions and the complexity of pluralization/singularization “magic.”
*   Data types and SQLite notes:
    *   ISBN stored as `TEXT` primary key (recognizing ISBN-10 vs ISBN-13 issues; kept simple with one field).
    *   `title` marked `NOT NULL`.
    *   Publication date stored as `TEXT` (SQLite has limited storage classes; date/time often stored as integer timestamps or strings).
    *   Avoided magic placeholder values (e.g., not using “1970” or “beginning of time” when unknown); preferred `NULL` when unknown.
*   Considered `genre` normalization:
    *   Noted books can have multiple genres; discussed that proper modeling would require tables and a join relation.
    *   Temporarily omitted genre to keep the demo manageable (later briefly reconsidered making a genre table).
*   Created `authors` table:
    *   `id` as integer primary key (surrogate), `name` as `TEXT NOT NULL`, `birth_date` as nullable `TEXT`.
    *   For unknown authors, decided to omit the relationship rather than create an “unknown author” row.
*   Created many-to-many join table `book_authors` (or similar):
    *   Fields: `isbn` (`TEXT NOT NULL`), `author_id` (`INTEGER NOT NULL`).
    *   Composite primary key on (`isbn`, `author_id`) to prevent duplicates.
    *   Foreign key constraints: `isbn` → `books(isbn)`, `author_id` → `authors(id)`.
*   Created `copies` table:
    *   Fields: `isbn` (`TEXT NOT NULL`), `copy_number` (`INTEGER NOT NULL`).
    *   Composite primary key (`isbn`, `copy_number`) to uniquely identify each copy; noted copy numbers may have gaps due to retired copies.
    *   Foreign key `isbn` → `books(isbn)`.
*   Created `patrons` table:
    *   `card_number` as integer primary key (with SQLite behavior: `INTEGER PRIMARY KEY` can auto-generate IDs if omitted on insert).
    *   `name` and `address` as required `TEXT NOT NULL`.
    *   `fine_balance` as a numeric type (e.g., `REAL` / double precision).
*   Created `lending` table:
    *   Fields: `card_number`, `isbn`, `copy_number`, `due_date` (stored as `TEXT`).
    *   Composite primary key initially proposed as (`card_number`, `isbn`, `copy_number`).
    *   Foreign key: `card_number` → `patrons(card_number)`.
    *   Composite foreign key: (`isbn`, `copy_number`) → `copies(isbn, copy_number)`.
*   Date storage advice when using text:
    *   Recommended ISO 8601 format (`YYYY-MM-DD` and optional `T` time, seconds, and timezone like `Z` for UTC) so lexicographic order matches chronological order.

### Querying the schema: multi-join path and grouping/counting

*   Framed a query goal: “Which authors is this patron currently reading?” / connect patrons to authors through current lendings.
*   Recognized join path constraints:
    *   `authors` is not directly connected to `patrons`; must traverse `lending` → `copies` → `books` → `book_authors` → `authors`.
*   Discussed `NATURAL JOIN` viability:
    *   Could `NATURAL JOIN` across tables where column names match, but not where they differ (e.g., `authors.id` vs `book_authors.author_id`).
    *   Used an explicit join condition for the `authors` link (via `authors.id = book_authors.author_id`).
*   Selected specific fields with aliasing to disambiguate duplicate column names:
    *   Selected `patrons.name` as patron name and `authors.name` as author name.
*   Noted duplicates and used `GROUP BY`:
    *   Grouped by both patron and author names to deduplicate patron-author pairs.
    *   Added `count(1)` to count how many records per group (e.g., multiple books by the same author checked out by the same patron).
*   Mentioned `COUNT(*)` vs `COUNT(1)`:
    *   Professor habitually uses `count(1)`; noted some systems may implement `count(*)` less efficiently (by materializing rows), though details vary.
