## Database Storage, Buffer Management, and Logging Beyond SQLite

*   The lecture steps back from `SQLite` as a case study to survey general database implementation choices, especially how databases store data on disk and interact with operating-system services.
*   A major theme is that databases often know their own access patterns better than the OS does, so default cache policies like approximate `LRU` can behave poorly for table scans and other planned query workloads.
*   The professor reviews practical `SQLite` configuration through `.sqliterc`, including cache sizing, lock timeouts, foreign-key enforcement, journaling modes, synchronous writes, temporary storage, and shell display settings.
*   The lecture then covers physical storage formats: fixed-length versus variable-length records, slotted pages, overflow storage for large objects, and why generous but explicit maximum field sizes still matter in secure systems.
*   It closes with file organizations, partitioning, catalogs, buffer managers, locking, write ordering, and the hardware realities behind logging and recovery, from spinning disks to `SSD`s and non-volatile write buffers.

## Lecture Outline

### Motivation: moving from `SQLite` specifics to general database implementation

*   The professor begins by noting that the class has spent a lot of time using `SQLite` as a case study and now wants to step back to cover more general implementation ideas.
*   The material comes from slides based on the Silberschatz textbook.
*   The goal is to look at common database storage structures, compare them with what the class has already seen in `SQLite`, and understand which ideas are general and which are specific to one system.

### Databases as files, raw devices, and OS interaction

*   Most databases are fundamentally collections of files.
*   `SQLite` follows this model in a very simple way by placing the main database in one file and using another file for the transaction log.
*   Some database systems go further and take over an entire disk partition or block device rather than living inside a normal file system.
*   This can be advantageous because databases often know much more about their own disk access patterns than ordinary applications do.
*   A recurring systems point is that operating-system policies are designed for broad, general-purpose use, while databases are extreme, specialized applications.

### Why OS cache policies can be bad for databases

*   The professor uses caching as the main example of a mismatch between OS policy and database needs.
*   Ordinary applications rely on the OS buffer cache and usually accept an approximate `LRU` policy, which assumes recently used pages are the best pages to keep.
*   For a full table scan larger than memory, `LRU` behaves badly:
    *   the scan fills the cache
    *   later pages evict earlier pages
    *   a second scan over the same table gets essentially no cache hits
    *   the scan also flushes other useful pages out of cache
*   Databases do this kind of workload frequently, unlike many ordinary applications.
*   A database also often knows ahead of time that such a scan is about to happen because query planning and execution planning are done in advance.

### Database-aware caching strategies

*   When thinking about what a database should keep in memory, the professor asks what would be most valuable in a `B+ tree`.
*   The answer is the upper levels of the tree, especially the root and other internal nodes.
*   Because `B+ tree`s have high fanout, the number of upper-level pages is small compared with the number of leaf pages.
*   If those upper levels remain cached, then most lookups only need disk access for the leaf page containing the actual row.
*   This makes traversals behave almost like constant-time operations plus one final access.
*   For a linear scan, a better strategy is often to avoid treating the scan as a signal that all of those pages should remain cached.
*   Possible responses include:
    *   effectively turning off caching for the scan
    *   using `MRU`-like behavior so the most recently read scan pages are the ones immediately evicted
    *   pinning valuable pages such as upper tree levels while letting scan pages cycle through a small working area
*   This is part of the motivation for systems that let the database control more of storage management directly, or even bypass parts of the file-system cache.

### `SQLite` defaults, cache sizing, and `.sqliterc`

*   The professor notes that `SQLite` was designed for convenience and simplicity, and that a single-file design works well for the overwhelmingly common case where databases are not very large.
*   `SQLite` can handle surprisingly large databases, but most real uses are much smaller than that limit.
*   He mentions that `SQLite` starts with a fairly small cache and that increasing cache size with `PRAGMA` settings is often a good idea.
*   He then shows his personal `.sqliterc` file from his home directory as a practical example of startup configuration.
*   One of the shell settings temporarily redirects output to `/dev/null` while startup settings are being applied so the shell does not print distracting feedback.
*   He sets the database encoding to `UTF-8`.
*   He sets `busy_timeout` so that if the database is locked, `SQLite` waits up to a specified number of milliseconds instead of immediately returning a busy error.
*   This matters because in the default journal and locking modes, a write transaction can lock the whole database and surprise users who expect smoother concurrent access.
*   He enables `foreign_keys`, since foreign-key enforcement is historically off by default in `SQLite`.
*   He uses `journal_mode = delete` for class purposes because it is simpler to reason about than `WAL`.
*   He keeps `locking_mode = normal`.
*   He discusses `synchronous` as a durability/performance tradeoff:
    *   stronger settings wait for data to actually be pushed to disk with calls like `sync` or `fsync`
    *   weaker settings can be faster but create a vulnerability window after the OS has accepted the write but before the storage device has really persisted it
*   He also notes a useful setting for temporary structures so that intermediate results can stay in memory rather than going to disk.
*   This matters because many queries create temporary tables or temporary indexes internally, and operations like `ORDER BY` or `DISTINCT` can be implemented by building temporary `B`-tree structures and then reading them back.
*   For shell display, he prefers `.mode table` for neat printing.
*   He also uses `.nullvalue` with a visible Unicode marker so that `NULL` values can be distinguished from empty output in query results.

### Rollback journals, `WAL`, and a backup tool

*   The discussion of `journal_mode` leads into a review of transaction properties, especially atomicity and durability from `ACID`.
*   In the older, simpler rollback-journal approach used by default `SQLite`, an undo log is created as a separate temporary file during a transaction.
*   The basic logic is:
    1.  before modifying a page, copy the original page into the undo log
    2.  apply the change to the original database page
    3.  if the transaction commits, delete the undo log
*   If a user explicitly rolls back, the same mechanism restores the old data.
*   If the program crashes or is killed mid-transaction, `SQLite` checks at startup for the undo log.
*   If it finds the journal and there is no active writer, it restores pages from the journal back into the database.
*   The implementation is intentionally simple: it does not try to infer which logged pages were actually modified successfully; it just copies everything in the journal back and then deletes the journal.
*   The professor then contrasts this with `WAL`, or write-ahead logging.
*   In `WAL`, the system appends intended changes and commit information to a log first, and only later applies those changes back into the main database file during checkpointing.
*   Compared with rollback journaling, `WAL` generally supports better concurrency and can perform better under contention.
*   The tradeoff is that the log file is persistent and the mechanism is a bit more complex.
*   For class use he prefers the simpler rollback-journal mode, but for production he suggests that `WAL` is often a better choice.
*   As an aside, he mentions `LiteStream`, an open-source tool for real-time `SQLite` backup.
*   `LiteStream` works only with `WAL` mode:
    *   it acts as another client of the database
    *   watches the growing `WAL`
    *   streams those changes to destinations such as `Amazon S3`, `Google Cloud Storage`, another machine, or another directory
    *   can support restoration to earlier points in time
*   He says he uses it for CodeGrinder because it provides data-center replication and backup at extremely low cost for small `SQLite` workloads.

### Fixed-length records and the origins of `CHAR` and `VARCHAR`

*   The lecture then shifts from logging to older physical record formats.
*   A very simple historical design is to store fixed-length records in a file.
*   Each row has a known width, so record `n` can be found by simple arithmetic and a direct seek.
*   This is easy to read and write but quite inflexible.
*   Problems include:
    *   every field needs a fixed or maximum size
    *   records can cross block boundaries
    *   deletion is awkward because removing one record creates holes or requires moving lots of data
*   The professor connects this to SQL string types such as `CHAR` and `VARCHAR`.
*   In older systems, these declarations reflected actual disk layout: a `VARCHAR` still had a maximum width reserved in the record, with shorter values padded somehow.
*   He says he has always disliked having to invent arbitrary maximum lengths for ordinary text like names.
*   In modern systems, and especially in `SQLite`, he generally prefers `TEXT`, since the engine already handles variable-length storage and `SQLite` does not meaningfully enforce `VARCHAR` limits anyway.

### Why arbitrary maximum lengths still matter

*   Although modern storage engines no longer need narrow fixed-width declarations for efficiency in the same way, the professor argues that arbitrary maximum lengths are still an important design principle.
*   He uses password handling as the example.
*   Best practice is:
    *   never store plaintext passwords
    *   store cryptographic hashes instead
    *   add a per-password salt so attackers cannot attack the entire database with one precomputed table
    *   use an intentionally slow password-hashing function so brute-force attacks become expensive
*   Some password-hashing schemes also deliberately consume lots of memory to limit parallel attack speed.
*   But this creates a new systems issue: if an attacker submits an absurdly large password input, the expensive hash computation itself can become a denial-of-service attack.
*   Therefore even fields that are logically “variable length” should often have generous but explicit maximum sizes.
*   His practical advice is to set limits higher than ordinary users need, but still make sure the system is tested and behaves correctly at those maxima.
*   He applies the same reasoning to other fields such as names: choose large but finite limits rather than pretending input is truly unbounded.

### Variable-length records and slotted pages

*   A better general storage approach is variable-length records.
*   Instead of fixing one width for all rows, the system stores records in order and includes metadata describing field sizes for each record.
*   This introduces per-record overhead, unlike fixed-length layouts where the schema alone tells the system how wide each record is.
*   The professor notes that this is similar to what the class saw in `SQLite`, where records have headers encoding type and length information.
*   He then points out a slotted-page structure and says it should look familiar because it is essentially what `SQLite` uses.
*   A slotted page has:
    *   a block header at the front
    *   a list of offsets or pointers to individual records
    *   the records themselves packed from the end of the page backward toward the middle
*   Because the slot array points to records indirectly, records can be moved within the page without changing all higher-level references.
*   In practice, systems often avoid aggressive compaction and only reorganize a page when space pressure makes it necessary.
*   He relates this back to `SQLite`, which tracks fragmented free space within a page and can compact when needed rather than constantly shifting bytes around.

### Large objects and where to store them

*   The professor briefly revisits how `SQLite` handles very large fields.
*   In a `B`-tree-based system, a record really needs to fit within a page-sized framework; otherwise the structure degrades and loses the properties the tree depends on.
*   One solution is overflow storage, where part of the value lives on another page linked from the main record.
*   `SQLite` uses this kind of mechanism for oversized values.
*   Other alternatives include:
    *   storing large objects as files in the file system and keeping only references in the database
    *   having the database itself manage those files so transaction semantics can still be coordinated
    *   breaking large objects into multiple tuples in a separate relation, as PostgreSQL does with `TOAST`
*   He emphasizes that relational databases can store large objects, but they are not usually optimized for workloads dominated by images or other big binary data.
*   Still, he stresses that one should not be dogmatic.
*   In CodeGrinder, he stores the files associated with assignments and submissions directly in the database because the traffic is low and the operational simplicity is worth it.
*   He says he originally stored them inside one large `JSON` object in a single column, but later moved to a separate files table linked by keys.
*   The key design choice is that he does not query by file contents; the database is just storing the files safely and transactionally.
*   For that workload, the convenience of “everything in one file” makes backup and management much simpler.

### File organizations: heap, sequential, `B+ tree`, and hashing

*   The lecture next surveys broader file-organization choices for storing records.
*   A `heap` organization means records are placed wherever space is available, with no meaningful physical order.
*   The professor describes this with the metaphor of a messy dorm room: find an empty spot and throw the item there.
*   Other possibilities include sequential files ordered by a search key and clustering organizations for related data.
*   `B+ tree`s are the main workhorse structure the class has already studied through `SQLite`.
*   He also mentions hash-based storage.
*   On-disk hashing is not just the ordinary in-memory hash-table structure reused unchanged.
*   Conventional memory hash tables rely on very cheap random access, which does not map cleanly onto disk.
*   Disk-oriented hash structures still use a hash of the key, but the actual packing of data and the access pattern have to be designed differently.
*   A major tradeoff is that hash-based structures give fast exact-key lookup but lose ordered access.
*   The professor explicitly agrees with the student summary:
    *   `B+ tree`s provide efficient lookup and support ordered scans
    *   hash tables provide efficient exact lookup but no useful order
*   He says that while some systems have supported both structures, `B+ tree`s are flexible enough that there is a clear reason `SQLite` uses them as its single core structure.

### Partitioning and the difficulty of scaling out

*   The next topic is partitioning, meaning splitting a relation into multiple physical pieces.
*   This can be done because the data is very large, but also because a particular workload is growing rapidly.
*   A common strategy is range partitioning, for example dividing a transactions table by year.
*   Logically it is still one relation, and the query engine understands the boundaries between partitions.
*   Partitioning is also a standard technique when data must be distributed across multiple machines.
*   The professor says partitioning has a reputation for being hard to manage, especially when spread across machines.
*   He explains why:
    *   a company may start on one database server
    *   then grow onto a few machines with a particular partitioning scheme
    *   later outgrow that scheme and need to repartition while the system is live
*   In practice, performance assumptions, locality assumptions, and even bits of application knowledge often become entangled with the original partition boundaries.
*   Moving from four nodes to eight or sixteen nodes while the system stays active can become a major multi-year migration project.
*   He notes that newer distributed systems still partition data, but often use more flexible strategies that are less visible to applications.
*   Partitioning can also be useful entirely within one machine, such as storing a single relation across multiple files to simplify free-space management.

### Catalogs and the data dictionary

*   The lecture then turns to metadata storage.
*   In `SQLite`, the class has effectively already seen the catalog when examining page 1 of the database.
*   After the 100-byte file header, page 1 begins the root `B`-tree that stores the definitions of tables and indexes.
*   `SQLite` keeps the original `CREATE` statements and reparses them when it needs to reconstruct schema information.
*   This is a very simple and compact design.
*   Larger database systems often use richer internal catalog tables.
*   Those tables describe columns, types, constraints, foreign keys, and other structural information about the rest of the database.
*   Because the catalogs are themselves relational, users and tools can query the database to discover its own schema and metadata.

### Buffer managers, one-copy semantics, and concurrency

*   The final major section concerns storage access rather than just storage layout.
*   A traditional relational database has a block or buffer manager sitting between higher-level components and the disk.
*   Higher layers request blocks from this manager rather than reading the disk directly.
*   If a requested block is already in memory, the manager returns the address of that in-memory copy.
*   A central invariant is that there should be only one cached copy of a given block at a time.
*   This matters because different transactions or different parts of one execution must all see the same current version rather than diverging private copies.
*   If a block is not already cached, the manager allocates space for it and may need to evict another block first.
*   If the evicted block is dirty, it may need to be written back before replacement.
*   The professor also discusses pinning or latching so that a page in active use cannot be evicted out from under the operation that is using it.
*   In response to a student question, he confirms that the buffer contents can differ from what is on disk until the system writes them back.
*   This becomes more complicated when multiple concurrent transactions are updating data, because the system must ensure that uncommitted changes do not leak to durable storage incorrectly.
*   With coarse page-level locking, one transaction may effectively block the whole page.
*   With finer-grained concurrency, the bookkeeping gets more complicated because committed and uncommitted changes may coexist in more intricate ways.
*   He notes that `SQLite`’s `WAL` mode provides more concurrency than the original coarse file-locking scheme.
*   He also reviews the common distinction between:
    *   shared locks for readers, which allow multiple concurrent readers
    *   exclusive locks for writers, which allow one writer and exclude other access

### Replacement policies and bad query patterns

*   The professor revisits cache replacement from the perspective of the buffer manager.
*   `LRU` is presented as the canonical replacement policy because it captures temporal locality, but exact `LRU` is expensive to implement and most systems use an approximation.
*   The ideal strategy would evict the block whose next use is farthest in the future, but ordinary systems do not know the future.
*   Databases can sometimes do better because execution plans reveal a lot about upcoming access order.
*   A full scan is again used as the classic case where `LRU` performs especially poorly by flushing the cache for little benefit.
*   Another bad case is a nested-loop join, where the inner scan can repeatedly destroy useful cached state before the next outer-loop iteration begins.
*   Better approaches include:
    *   freeing scan pages as soon as they have been consumed
    *   using `MRU`-style behavior for certain scans
    *   protecting especially valuable pages such as upper `B+ tree` levels from eviction

### Write ordering, logs, and hardware realities

*   The lecture closes by discussing the fact that the operating system may reorder writes.
*   Even if an application issues writes in a careful logical order, the OS may send them to storage in a different order for efficiency.
*   This is another reason logs are so useful: they reduce correctness to a simpler story about making the log durable and replaying or undoing from there.
*   A strong design principle is that recovery procedures should be simple.
*   An even stronger principle is that, if possible, normal operation and recovery should use the same mechanisms so that recovery code is regularly exercised rather than only during rare failures.
*   A student asks why the OS would reorder writes, and the professor explains in terms of spinning hard drives.
*   Disk performance is dominated by:
    *   seek latency, the time to physically move the heads
    *   rotational latency, the time to wait for the desired sector to rotate under the head
*   Because of those costs, request schedulers often use strategies like the elevator algorithm, sweeping across the disk by cylinder so that many nearby requests are served in one pass.
*   That improves throughput even though it changes the order of individual requests.
*   He then contrasts this with `SSD`s.
*   `SSD`s do not have mechanical seek and rotation, but they still hide complex remapping behavior from the database.
*   Logical block writes may be redirected elsewhere internally, with wear leveling spreading writes across flash cells rather than repeatedly updating the same physical location.
*   So even on modern storage, the physical reality is more complicated than “the DB wrote block `n`, so block `n` was updated in place.”

### Specialized write buffers and concluding remarks

*   As a final aside, the professor mentions non-volatile write-buffer hardware.
*   Older high-end designs might use battery-backed `DRAM` or similar `NVRAM` so that log records could be written at memory speed while still surviving power loss.
*   Some systems also combined `DRAM`, flash, and capacitors so memory contents could be preserved into flash on power failure.
*   This kind of hardware is especially attractive for journaling because the database can make the log durable quickly and defer slower writes to the main data area.
*   He notes that such tricks are more associated with specialized or high-end database deployments than with systems like `SQLite`, and that modern distributed approaches often address performance and durability in different ways.
*   The lecture ends here because time is up.
