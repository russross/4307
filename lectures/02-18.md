### Anatomy of a recursive CTE for management chains

*   Introduced the structure of a recursive CTE (shown in a “create view” example, but used as a normal `SELECT` as well)
*   Described the CTE header as defining a temporary result relation (e.g., `boss_path`) with columns like `cn` and `path`
*   Explained the intended contents:
    *   `cn`: the identifier for the person
    *   `path`: a string representation of the management chain, built by concatenating names/identifiers with a separator (e.g., `' > '`)
*   Broke the recursive CTE into two required components separated by `UNION`/`UNION ALL`:
    1.  Base case (“setup step”): run once to seed the recursion
    2.  Recursive step: repeatedly executed, referencing the CTE being built
*   Explained the evaluation behavior:
    *   Start with the base-case rows
    *   Re-run the recursive query using the accumulated rows so far
    *   Keep adding new rows until a fixed point (no new rows) is reached (“converges”)
*   Commented on duplicates:
    *   With `UNION`, duplicates collapse; with `UNION ALL`, duplicates can accumulate unless constrained
    *   Emphasized the conceptual idea that results are “merged in” each round

### Manually simulating recursion with temporary tables (debugging approach)

*   Recommended not writing the full recursive query “all at once”; instead, reproduce the steps to test incrementally (similar to a homework-style debugging method)
*   Created a temporary table `A` to store base-case results (columns like `cn` and `path`, with minimal/no types for demonstration)
*   Inserted into `A` the base-case query: all users with no manager
*   Verified by running `SELECT * FROM A` (showing just the top-level manager)
*   Prepared to simulate the recursive step using another table `B`

    *   Built a new `path` by concatenating the current user with the prior `path` (using the `' > '` separator)
    *   Joined with a condition equivalent to `users.manager = A.cn` (i.e., find direct reports of the current level)
*   Noted and corrected naming/column mistakes during the live build:
    *   Adjusted table definitions to match the intended CTE columns (`cn`, `path`)
    *   Recreated `A`/`B` after dropping them to get a clean run
*   Inserted first recursive-step results into `B` and inspected:
    *   `A` contained the root (no-manager) row(s)
    *   `B` contained all users who directly report to the root, with their computed `path`

### Repeating the step to grow the hierarchy level-by-level

*   Explained the iterative idea: each round joins `users` against the accumulated results to find the next level of reports
*   Demonstrated successive iterations by creating additional tables and populating each by joining against the previous:
    *   Created `C` by joining against `B` (producing “three-level hierarchy” paths)
    *   Created `D` by joining against `C` (four-level hierarchies)
    *   Created `E` by joining against `D` (five-level hierarchies), with a brief correction after noticing a self-join/mistake
    *   Created `F` by joining against `E` (six-level hierarchies)
    *   Created `G` by joining against `F`, which returned no rows (bottom reached)
*   Interpreted the termination condition: an iteration that produces an empty result indicates no further descendants exist
*   Connected back to what the recursive CTE does automatically:
    *   The CTE effectively performs these repeated “join and extend path” steps internally
    *   The recursion stops when no new rows are produced
*   Observed what the final result corresponds to in the manual simulation:
    *   A conceptual final output like `A UNION B UNION C UNION D UNION E UNION F` (union of all levels) gives the full set of computed paths

### Using the recursive result to “hydrate” final output

*   Pointed out that recursive construction often uses minimal columns (e.g., IDs) for efficiency
*   Described a common pattern:
    1.  Build the recursive intermediate relation (e.g., `boss_path`) using only what’s needed for recursion
    2.  Join the recursive result back to the base tables (e.g., `users`) to fetch full details (names, titles, etc.)

### Q&A: can the recursive result be formatted as a tree?

*   Question: whether the “chain” result could be treated as a tree and displayed/converted directly
*   Response:
    *   A tree is inherently 2D; the `path` representation traces a linear chain
    *   You could compute “all descendants,” and possibly format output for display
    *   Mentioned the possibility of creating a nested structure (e.g., JSON hierarchy), but not as a straightforward built-in transformation in this context

### Main recursive task: Bacon numbers on the IMDb graph

*   Defined the Bacon number rules for the assignment:
    *   Kevin Bacon has Bacon number `0`
    *   Anyone who worked directly with him has `1`
    *   Anyone who worked with someone at `1` has `2`, etc.
*   Identified a practical issue with recursive graph expansion:
    *   People can be rediscovered via longer paths, creating multiple Bacon numbers for the same person (including Kevin Bacon appearing again at 2, 3, etc.)
    *   Without careful “visited set” filtering, recursion may not converge naturally
*   Described constraints/limitations encountered (SQLite-related):
    *   Difficulty expressing “exclude nodes already seen” due to restrictions on how the recursive CTE can reference the growing result
*   Workaround imposed for the assignment:
    *   Cap the recursion depth (Bacon number) at `4`
    *   Add a `WHERE` filter in the recursive step to stop expanding beyond 4, ensuring convergence
*   Performance guidance:
    *   Recursive CTE performance is sensitive to query structure
    *   Keep the recursive working table minimal (e.g., store only `person_id` and `bacon_number`, not names)
    *   Defer joining to `people` (for names) until the final outer query

### Suggested incremental testing strategy for the Bacon-number query

*   Recommended the same stepwise approach used in the management-chain example:
    *   First, write a query that finds Kevin Bacon in `people` by name (don’t hardcode IDs)
    *   Next, write a query that finds all people at Bacon number `1` (direct co-crew on titles)
    *   Optionally insert these into a temporary table and join against it to compute Bacon number `2`
*   Warned about duplicates and multiple paths:
    *   The recursive build can generate multiple entries for the same `person_id` with different candidate numbers
    *   The final output should select the minimum Bacon number per person
        *   Use `GROUP BY person_id` with `MIN(bacon_number)` (or equivalent) in the final stage

### Minimum schema requirements for the social network

*   Required baseline tables/features:
    *   `users`: one human per user; store email and possibly other info
    *   `accounts`: separate from users; a user can have multiple accounts; an account belongs to exactly one user (one-to-many)
    *   `followers`: directed follow relationship; mutual follow is represented as two rows
    *   `posts`: text content posted by an account
        *   Must include at least a timestamp along with the text
*   Additional requirements:
    *   At least two more tables (join tables do not count toward the “two more tables” requirement)
    *   Examples of additional tables: likes, blocks, reports, DMs, etc.

### Feature ideas for extending the social network

*   Likes
    *   Track count of likes per post
    *   Use likes to create alternative feed views (e.g., most-liked posts from accounts you follow in the last 24 hours)
*   Reporting posts
    *   If a user reports a post, hide it from that user
    *   If enough reports accumulate, hide it more broadly or exclude it from “popular” views
*   Blocking
    *   Blocked accounts cannot follow you, see your posts, DM you, or otherwise interact
*   Direct messages (DMs)
    *   Model message threads/conversations between two accounts
*   `@mentions` and hashtags
    *   Mentioning may generate notifications or make content appear in a user’s view even without a follow
    *   Suggested implementation approach: parse mentions/hashtags at insert time and store extracted references in additional tables
*   Boosting/promoted posts
    *   Add a notion of boosted visibility, similar to algorithmic feeds
    *   Mentioned using randomness (e.g., a `random()`-based filter) to probabilistically surface boosted posts
*   Muting
    *   Continue following an account but exclude their posts from your feed
*   Shadow banning
    *   Let a user believe their content is visible while others don’t see it (compared to “cheater island” approaches in games)
