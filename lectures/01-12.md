## CS 4307 Database Systems: SQL Crash Course, Single-Table Queries, and Why “Relational” Matters

*   Course roadmap: first half focuses on using relational databases via SQL; second half focuses on how database systems are implemented internally
*   Practical SQL basics using SQLBolt and SQLite: tables/rows/columns, `SELECT`/`FROM`, and narrowing results by columns vs. rows
*   Filtering and matching: `WHERE` with numeric operators, set-style operators (`IN`, `BETWEEN`), and string matching with `LIKE` (including wildcards)
*   Result shaping: handling duplicates with `DISTINCT`, sorting with `ORDER BY`, and paging with `LIMIT`/`OFFSET`, plus brief notes on performance and interface design
*   Conceptual pivot to relational design: tables as *relations*, normalization to reduce redundancy, and contrast with denormalized / key-value approaches (Twitter-style feed caching)

### SQL Query Structure (Single Table): `SELECT` and `FROM`

*   Minimal query pattern introduced: `SELECT ... FROM ...`
    *   `FROM` identifies which table(s) to pull data from
    *   `SELECT` chooses which columns to return
*   The `*` wildcard to select all columns (`SELECT * FROM ...`)
*   SQLBolt Pixar movies example begins
    *   Task flow: start with `SELECT * FROM movies` then narrow columns
*   Exercises on projecting columns (narrowing “horizontally”)
    *   Return only titles: use `SELECT title FROM movies`
    *   Return only directors: `SELECT director FROM movies` (observing duplicates)
    *   Return multiple columns: comma-separated list (e.g., `title`, `director`)
    *   Return all info again: `SELECT * ...`
*   Conceptual aside: pure relational model uses *sets* (no duplicates), but SQL often returns *bags/multisets* (duplicates can appear)

### String Matching and SQL Dialect Differences

*   String equality uses single `=` (no `==` because there is no assignment operator in SQL expressions)
*   “Not equal” forms and standardization
    *   `<>` emphasized as the long-standing/standard SQL “not equal”
    *   `!=` described as widely implemented but *not* standard SQL
    *   Instructor briefly looks it up to confirm: `!=` is nonstandard (though common)
*   SQL standard vs. implementations
    *   There is a SQL standard, but no system fully matches it
    *   Systems vary in behavior and supported syntax (PostgreSQL cited as relatively good)
*   Pattern matching with `LIKE`
    *   `LIKE` described as commonly case-insensitive by default in many systems (including SQLite), with options to change behavior
    *   Wildcards:
        *   `%` matches any sequence of characters
        *   `_` matches a single character
*   Quoting strings vs. identifiers
    *   Correct practice: string literals in single quotes (e.g., `'Toy Story%'`)
    *   Double quotes are for identifiers; SQLite may accept more, but “don’t do that” if aiming for correctness

### De-duplication and Result Shaping: `DISTINCT`, `ORDER BY`, `LIMIT`, `OFFSET`

*   `DISTINCT` to remove duplicates in results
    *   Motivated by duplicated director names returned earlier
    *   Clarification: `DISTINCT` applies to the *entire result rows*, not just “the next column”
*   Sorting results with `ORDER BY`
    *   Default is ascending; can specify `ASC` or `DESC`
    *   Multi-key sorting: `ORDER BY col1, col2` for tie-breaking
    *   Performance note: sorting forces extra work; be judicious with large datasets
*   Limiting/paging with `LIMIT` and `OFFSET`
    *   `LIMIT n` returns only the first `n` rows of the result set
    *   `LIMIT n OFFSET m` skips the first `m` rows then returns `n`
    *   Typical use case: paging in web interfaces (“next page”)
*   Instructor aside: frustration with overly small paging defaults in campus systems (e.g., showing only 20 results)

### What “Relational” Means: Tables as Relations and Links Between Tables

*   The key pivot: multi-table queries are where the “relational” concept becomes real
*   Technical terminology: a table is a *relation*
*   Design intent: each table stores one kind of information in a “neutral” / natural form
    *   Not organized primarily around a specific access pattern
*   Relationships between tables via referencing columns
    *   Example: `people` table and `homes` table
        *   `homes` includes a column referencing the owning person
        *   Keep “home attributes” (bedrooms, lot size, etc.) separate from “person attributes”
*   Another example: music collection schema
    *   Tables for artists, albums, tracks
    *   Relational approach avoids forcing a single hierarchy
    *   Hierarchical model breaks down with collaborations and compilations (albums with multiple artists)
*   Preview term: normalization (formal methods for structuring tables to avoid redundancy)

### Normalization Motivation: Avoid Redundancy and Inconsistency

*   Example: duplicating artist/band metadata on every album row
    *   Inefficiency: repeated storage
    *   Correction problem: fixing a mistake requires updating many rows
    *   Consistency problem: accidental mismatches/typos across copies
*   Spreadsheet analogy and institutional example
    *   Scheduling spreadsheets with “way too many columns” (~30 columns)
    *   Repeated building/room names create misspelling opportunities
    *   Room capacity (e.g., “Smith 108 has capacity 32”) should be stored once and referenced, not repeated
*   Big-picture framing
    *   Normalize data into neutral tables with relationships
    *   Use queries (not data duplication) to assemble the specific view needed for a task
    *   Enables new queries later without restructuring stored data

### Querying Across Many Tables: Flexibility as a Core Benefit

*   Instructor gives a conceptual example query (without implementing it)
    *   “Find everybody who has ever taken a class with a particular student”
    *   Requires traversing multiple tables (student → classes → other students, etc.)
    *   Emphasizes multi-step navigation over a graph of relations
*   Key point: relational databases separate *data representation* from *access patterns*
    *   Ad hoc querying becomes possible without redesigning the storage layout

### Contrast: Non-Relational / Key-Value / Document Stores (e.g., MongoDB)

*   Mentions non-relational databases where a key maps to a value (often a JSON-like document)
*   These systems can be attractive in some scenarios, but illustrate the opposite design philosophy

### Denormalization at Scale: Twitter Timeline Example

*   Twitter (pre-X) framed as an instructive case
    *   Relational model is “beautiful and simple” conceptually but can fail at massive scale for certain access patterns
*   Normalized relational design sketch
    *   Users table; posts table referencing users
    *   Followers table representing follow relationships
    *   Main query: “show me the 20 most recent posts by people I follow” (pre-algorithmic feed)
*   Scaling problem: too expensive to compute feeds via joins at extreme scale
*   Denormalized approach described
    *   Maintain a cached timeline per active user: a list of recent post IDs from followed accounts
    *   When someone posts, push that post ID into each follower’s cached timeline
    *   Observation: reading happens far more often than posting (optimize for reads)
    *   Serving the timeline becomes a single lookup + “hydration” (fetch the content by ID)
*   Celebrity edge case (Lady Gaga example)
    *   Massive follower counts make push-based fan-out too expensive
    *   Hybrid strategy: normal users get fan-out-on-write; top celebrities handled differently
        *   Keep separate “check these accounts” logic rather than pushing to every follower
*   Lesson tied back to relational theory
    *   Denormalization = structuring storage around frequent queries
    *   Relational default = store neutrally, then rely on a sophisticated query engine + indexing to execute many queries efficiently
*   Instructor perspective: relational query engines are an engineering marvel (efficient graph traversal and composition)
